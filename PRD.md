## 1. 개요

전략팀 대시보드는 프로젝트 기반 조직의 인력·일정·휴가·가동률을 통합 관리한다. 간트 타임라인, 휴가 자동/수동 산정, Utilization 대시보드, 개인 CV, Engagement 검색을 제공하며 권한을 RLS로 강제한다.

## 2. 역할

| 역할 | 접근 |
|---|---|
| admin | 전체 편집 + 계정·권한·설정·백업 + Admin·Migration 메뉴(admin 전용) |
| editor | 전체 데이터 열람·편집 |
| **assistant** | *(v2.11 신설)* **지정된 화면(연차 관리 포함)에 한해 열람 전용** — 편집 컨트롤 비활성, 서버 RLS도 조회(SELECT)만 허용 |
| viewer | editor와 동일 화면을 보되 **전면 읽기 전용**. CV Generator·Leave는 **본인 정보만**, Admin·Migration 비노출 (§6.3). **Pipeline 비노출·Confidential 필드 마스킹은 assistant와 공통 규칙(W-7) 적용(v2.101)** |

| 테이블 | 주요 컬럼 |

> 입력 형식: Engagement Code는 기본 `E-00000000`(E- + 숫자 8자리) 형식이며, **예외적으로 `C000000AA` 형식(C + 숫자 6자리 + 영문 대문자 2자리)과 `I-00000000` 형식(I- + 숫자 8자리, 내부코드)도 허용**한다(세 형식 중 하나만 만족하면 유효). LPN 숫자 5자리 `00000`. 계정↔인력 매칭은 `profiles.person_id`=`people.id`로만(LPN 미사용).

> **holidays 데이터 소스**: 공휴일(holidays) 데이터는 외부 공공 API(특일 정보 등)에서 연도별로 조회해 채운다. **조회 가능 범위는 최소 2022년부터 현재+1년까지 지원**해야 한다(과거 프로젝트휴가·주말대체 재계산, 퇴사 정산의 소급 계산 등에 2022년 이후 공휴일 데이터가 필요하므로). 현재 2025년부터만 조회되고 있다면, 사용 중인 API가 과거 연도 조회를 지원하는지 확인 후 범위를 확장한다(대부분의 공공 특일 정보 API는 연도 파라미터로 과거 연도도 조회 가능).
> **동기화 버튼 동작(버그 수정 + 단순화)**: 공휴일 화면의 버튼을 **"동기화" 하나로 단순화**한다 — 별도의 "과거 데이터 채우기" 버튼은 불필요하므로 **제거**한다. "동기화" 버튼을 클릭하면 **2022년부터 현재 연도+1년까지의 모든 연도**를 API에서 조회해 holidays 테이블에 일괄 반영한다(현재는 현재 연도+차년도 2개년만 반영되는 제한된 동작으로 되어 있음). **동기화 시 기존에 수동으로 입력해 둔 공휴일은 덮어써지거나 삭제되어도 무방**하다(API 조회 결과로 전체 교체). 동기화 완료 후 결과(반영된 연도 수·공휴일 건수)를 요약해 보여주고 audit_log에 기록한다.
> **API 호출 속도 제한 대응(버그 수정)**: 2022~현재+1년 전체를 동기화하면 월별/연도별 API 호출 건수가 많아져 **외부 API의 rate limit(HTTP 429)에 걸려 일부 월이 누락**될 수 있다. 이를 방지하기 위해 ① 각 API 호출 사이에 지연(예: 200~500ms)을 두어 순차 호출하고, ② HTTP 429 응답을 받으면 지수 백오프로 재시도(예: 최대 3~5회)한 뒤에도 실패하면 그 달만 오류로 기록하고 나머지는 계속 진행한다. ③ 오류가 발생한 월은 동기화 이력에 남기되, **같은 "동기화" 버튼을 다시 누르면 이미 성공한 월은 재조회하지 않고(멱등) 실패했던 월만 재시도**되도록 하여 재시도가 쉽도록 한다.

## 5. 기능 요구사항

### 5.1 실행취소 / 재실행
- Ctrl+Z=실행취소, Ctrl+Y(및 Ctrl+Shift+Z)=재실행. 최대 10단계. 대상: 배정·작업항목·인력·휴가 수동조정. 서버 역연산, 불가 시 안전 실패. editor/admin만. Undo/Redo 버튼.

### 5.2 타임라인 시각화
| ID | 요구사항 |
|---|---|
| T-1 | 인력별/작업별 뷰. 월→주→일 자동 전환. 모든 날짜 MM/DD. 반응형. 막대 라벨 폭 확장 + 호버 툴팁(작업명·고객사·기간, 인력별 뷰는 인력명). **인력별 뷰의 project·proposal·pipeline 막대 라벨 형식(신설)**: 막대 위에 프로젝트명만 표시하던 것을 **"[고객사명] 프로젝트명"** 형식으로 바꾼다(예: "[화장품 브랜드사] 리뉴얼 전략 수립"). 고객사명이 없는 경우(client 미입력)는 대괄호 없이 프로젝트명만 표시한다. **작업별 뷰의 워크아이템 라벨 칸 폭 확장(버그 수정)**: 작업별 뷰 좌측에 고정 표시되는 워크아이템명 칸의 폭이 좁아 제목이 길면 의도치 않게 줄바꿈되어 가독성이 떨어진다 — 이 칸의 폭을 더 넓게 확장한다(필요 시 사용자가 드래그로 조절 가능한 리사이즈 핸들을 추가해도 좋다). |

| T-3 | 작업 행은 그 작업 자체의 일정(span)만 표시(인원 막대 미중첩). 인원별 일정은 드롭다운으로. |

Error Boundary는 안전망으로 유지하되, 정상 흐름에서는 뜨지 않아야 한다. **⑤ 범위 밖 항목의 레인 배정 제외(재확인, v2.95)**: 조회 범위와 전혀 겹치지 않는 배정(종료 < 범위 시작 또는 시작 > 범위 종료)은 렌더뿐 아니라 레인 배정 대상에서도 제외되어야 한다 — v2.95 코드 검토 결과 T-22(레인 패킹, v2.94)가 이미 정확히 이 조건(`!(end < viewStart \|\| start > viewEnd)`)으로 후보를 걸러내고 있음을 확인했다(추가 수정 불요). **⑥ [⑤ 결론 번복 — 런타임에서 여전히 재현됨, 버그 수정, v2.96]**: ⑤에서 "코드상 이미 제외되므로 수정 불요"로 판단했으나, **실제 화면(Work Item 탭, FY25·FY26 조회)에서는 조회기간 이전에 종료된 workitem이 여전히 레인을 배정받고 조회기간 첫날 위치에 얇은 막대(슬리버)로 칠해진다** — 코드 리딩 결론이 런타임과 어긋났던 사례. **원인 확정(실제 DB 데이터로 재현)**: FY25 조회(viewportStart=2024-07-01) 기준 그보다 먼저 종료된 `status='open'` work_item이 8건 존재하며(예: `188ef602-…` "오디오 앰프 IC 제조사 CDD", 2023-06-12~2023-07-07) — Work Item 탭의 행 필터(`fw`)가 Closed 여부만 걸렀지 조회 범위와의 겹침은 전혀 안 봤으므로 이 8건이 항상 행으로 나타났고, 그 배경 밴드를 그리는 `WorkItemBand`는 `AssignmentBar`와 달리 완전 비겹침(`end < viewStart`) 케이스에 대한 조기 종료가 아예 없었다. 대신 `visStart=max(start,viewStart)`, `visEnd=min(end,viewEnd)`로 클램프한 뒤 `w = max((max(visEnd,visStart)-visStart+1)×dayWidth, 4)`로 폭을 구했는데, `end<viewStart`이면 `visEnd<visStart`가 되어 음수 폭이 나와야 할 자리를 `Math.max(visEnd,visStart)`가 `visStart`로 되돌리면서 `w`가 `dayWidth`(1일치)로 떨어지고 `x=(visStart-viewStart)×dayWidth=0`이 되어 **조회 범위 첫날에 1일짜리 슬리버가 그려지는** 부작용을 냈다(증상과 정확히 일치, 근본 원인 확정). 날짜 타입 불일치(문자열/Date/타임존)나 null 날짜 fallback은 확인 결과 해당 없음 — `dateToNum()`으로 양쪽 다 정수 day-number로 정규화되고, `start`/`end_date`는 NOT NULL 컬럼이라 null이 될 수 없다. **조치**: ① `AssignmentBar`·`WorkItemBand`·레인 패킹(T-22)이 공유하는 `overlapsViewport(start, end, viewStart, viewEnd)` 유틸을 신설(원본 언클램프 날짜 비교, `end>=viewStart && start<=viewEnd`). ② `WorkItemBand`에 이 조건이 거짓이면 클램프 계산 전에 `return null`하는 조기 종료를 추가(모든 훅 호출 이후에 위치시켜 v2.91의 React #300 재발을 피함) — 이러면 min-width 계산 자체가 비겹침 케이스에 도달하지 않으므로 슬리버가 원천적으로 사라진다. ③ Work Item 탭의 행 필터(`fw`)에도 같은 `overlapsViewport` 조건을 추가해 조회 범위와 안 겹치는 work_item은 행 자체가 나타나지 않게 함. 실제 데이터(위 8건)로 재확인 완료: FY25·FY26 조회에서 모두 배제됨. |

| **E-5** | **Workitem 기간 자동 확장**: 소속 인력의 투입 **종료일이 workitem 종료일(end_date)보다 늦어지면 workitem의 종료일을 그 인력의 종료일로 자동 연장**한다. 반대로 인력의 투입 **시작일이 workitem 시작일(start)보다 앞서면 workitem의 시작일을 그 인력의 시작일로 자동 앞당긴다**(본 프로젝트 시작 main_start는 그대로 두되 start보다 앞설 수 없다는 무결성 유지). 배정 생성·이동·리사이즈·일괄 조작 어느 경로로든 동일 적용. Closed 작업은 잠금 규칙상 배정 변경 자체가 불가하므로 해당 없음. |
| **E-6** | **배정 리사이즈/기간 변경 시 재검증(버그 수정)**: 배정을 생성한 뒤 드래그로 리사이즈하거나 기간을 늘려 특별휴가 등 잔여 검증 대상 휴가 유형의 사용일수가 증가하는 경우, **신규 생성 시와 동일하게 LV-6(특별휴가 잔여 검증)을 다시 수행**한다(현재는 최초 생성 시에만 검증하고 이후 리사이즈로 늘어난 기간은 검증 없이 저장되는 버그가 있음). 검증에 실패하면 리사이즈를 반려하고 원래 크기로 되돌리며, 부족분을 안내한다. 서버(RPC/트리거)에서도 동일 검증을 강제한다. |
| E-4 | viewer는 모든 편집 컨트롤 비활성(서버 RLS로도 차단). |

| ID | 요구사항 |

| **P-1** | **재직 여부 실시간 파생(버그 수정)**: 재직 여부는 저장되는 값이 아니라 **조회 시점의 오늘 날짜(UTC 일 단위)와 hire_date·termination_date를 매번 비교해 동적으로 계산**한다. 규칙: 오늘 < hire_date → **입사 예정(비재직)**, hire_date ≤ 오늘 이고 (termination_date 없음 또는 오늘 < termination_date) → **재직중**, termination_date ≤ 오늘 → **퇴사(비재직)**. **현재는 레코드를 새로 불러와 저장하는 시점에만 termination_date를 비교해 분류하고 있어, 퇴사일이 지나도 재저장 전까지 재직자로 남거나 입사일이 안 지났는데도 재직자로 표시되는 버그가 있다.** 이 값을 저장·캐시하지 말고 매 조회·목록·필터·검색(E-2 인력 선택, LV-2/AL-2a 재직중 필터, §8 Utilization 등)마다 실시간 재계산해 사용한다. |
| **P-2** | **재직 상태 3단 구분을 전 필터에 반영**: 기존에는 "재직/퇴직" 2단으로만 구분하던 것을 **재직중 / 입사예정 / 퇴사** 3단으로 확장한다. 인력 관리(§5.4) 목록의 재직 여부 표시, 그리고 **재직중 필터를 제공하는 모든 화면(LV-2, AL-2a, CV Generator §5.9 등 공용 필터 컴포넌트를 쓰는 모든 곳)**에서 이 3단 구분을 선택할 수 있게 한다(예: 재직중/입사예정/퇴사 중 하나 이상 다중 선택). P-1의 판정 기준(오늘 날짜 실시간 비교)을 그대로 사용한다. |

제외)에서 강제한다. **[v2.101 진단 결과] 이 조건은 실제로는 이미 `assistant`에게도 정확히 적용되어 있었다** — 라이브 RLS 정책을 직접 조회하고 실제 assistant 세션으로 SQL을 실행해 pipeline 행·해당 assignments가 0건으로 안 보임을 확인했다(서버 정책이 애초에 `my_role() IN ('editor','admin')`만 예외로 두는 방식이라 viewer·assistant를 구조적으로 동일하게 취급하고 있었음 — 서버 변경 불필요). **② Confidential 마스킹(필드 단위, 행은 항상 노출)**: `confidential=true`인 작업항목(project/proposal — pipeline은 ①로 이미 제외되므로 해당 없음)은 **admin·editor에게는 전체 필드가 그대로 노출**된다(마스킹 없음). **viewer·assistant에게는 다음과 같이 필드 단위로만 마스킹**한다 — **마스킹 대상(→ "(Confidential)"로 대체 표시): 이름/제목(name), 고객사명(client), Description, 해시태그(hashtags).** **마스킹하지 않는 필드(그대로 노출): 기간(전체 시작·본 프로젝트 시작·종료일), 참여 인력(배정된 인력 목록과 그들의 수행 기간), 유형(project/proposal), 상태(Open/Closed).** **[v2.101 진단 결과] `work_items_safe` 뷰도 이미 `COALESCE(my_role(),'') <> ALL(ARRAY['admin','editor'])` 조건으로 마스킹하고 있어 assistant가 처음부터 viewer와 동일하게 마스킹 대상이었다** — confidential 항목을 실제 assistant 세션으로 조회해 name="(비공개)"·client/description=null·hashtags=[]이고 기간·상태는 그대로임을 SQL로 확인했다(서버 뷰도 변경 불필요). **타임라인 간트차트에서는 confidential 작업항목의 막대 자체가 viewer·assistant에게도 정상적으로 렌더링된다**(행이 사라지거나 숨겨지는 것이 아님) — 막대의 라벨(제목)만 "(Confidential)"로 대체되고, 막대의 위치·길이(기간)와 그 막대에 배정된 인력 표시는 평소와 동일하다(코드 확인 결과 이미 그렇게 동작 — 행을 제외하는 지점 없음). **③ [v2.101 실제로 발견·수정한 클라이언트 결함 2건]** 서버는 이미 올바른데도, 일부 클라이언트 코드가 서버가 이미 마스킹해 내려준 값을 그대로 쓰지 않고 **`profile.global_role === 'viewer'`로 직접 재분기**해 assistant를 빠뜨리고 있었다(오늘 당장은 서버 마스킹이 이중 안전망 역할을 해 실제 유출은 없었지만, 향후 이 필드가 마스킹되지 않은 경로로 바뀌면 그대로 새는 취약한 코드였다) — **(a) `DashboardPage.tsx`**의 Kick-off/종료 목록이 `isViewer && wi.confidential`로만 이름·고객사 마스킹 여부를 판단하던 것을 `!canEdit('global')` 기반의 role-무관 `hideConfidential`로 교체. **(b) `TimelineView.tsx`의 통합검색**이 `description` 마스킹 조건에 confidential이 아니라 **엉뚱하게 pipeline 여부(`isPipelineViewer`)**를 쓰고 있었고(사실상 죽은 코드 — pipeline 행은 애초에 RLS로 안 옴), `hashtags`는 마스킹 가드 없이 그대로 검색 필드에 포함되고 있었다 — 둘 다 `isConf(wi)`(confidential && !globalEdit) 기준으로 통일. **④ 검색 매칭 제외**: 마스킹된 필드(이름·고객사·Description·해시태그)는 viewer·assistant 세션에서 검색어 매칭 대상에서 제외한다(마스킹된 원문으로 검색해도 결과에 걸리지 않음) — 전역 검색(G-4)은 `wi.name !== '(비공개)'`로 이미 항목 자체를 결과에서 제외하고 있었고, Engagement 검색(§5.8)은 서버 마스킹 값만 사용해 원천적으로 안전했다. |

### 5.9 개인 CV (CV Generator)
- 인력별 수행 작업항목(Engagement No., Client, 본인 수행기간=본 프로젝트 한정, 해시태그). Open/Closed 무관. 수행 프로젝트만 필터(기본). Pipeline 제외(**viewer·assistant 기준**, W-7). **이름 검색·필터 제공.** **viewer·assistant가 타인의 CV를 조회하는 경우 confidential 항목은 필드 단위 마스킹(W-7)이 적용**된다(Client·해시태그 등은 "(Confidential)", 수행기간은 그대로 노출).
- viewer는 본인 CV만 조회·다운로드.
- **V-5 (신설) CV 내 Engagement 부분 검색/필터**: 특정 인력의 CV를 연 상태에서, 검색어(§5.11a G-5의 AND/OR 문법 지원)로 그 사람이 수행한 Engagement 목록을 프로젝트명·고객사명·해시태그·Description 기준으로 필터링해 **일부 Engagement만 선택적으로 화면에 표시**할 수 있다(예: 특정 산업·클라이언트 관련 프로젝트만 뽑아서 보여주고 싶을 때). HTML 다운로드 시에도 **"현재 필터링된 항목만 내보내기" 옵션**을 제공하여, 전체 이력이 아니라 검색으로 좁힌 일부만 담은 CV를 생성할 수 있게 한다(옵션을 끄면 기존처럼 전체 이력 포함).
- **V-19 (신설, v2.105) 목록 항목 클릭 시 우측 상세 패널 — Engagement 검색과 동일 컴포넌트 재사용**: CV Generator에서 인력의 수행 Engagement 목록 중 항목을 클릭하면, Engagement 검색(§5.8)에서 결과 항목을 클릭했을 때와 **완전히 동일한 우측 상세 패널 컴포넌트**(`WorkItemDetailPanel`, HashtagPage.tsx의 기존 `DetailPanel`을 `src/features/workitems/`로 추출해 두 화면이 공유)가 화면 우측에 나타난다(목록은 좁은 열로 축소되고 우측에 상세가 열리는 master-detail 레이아웃, 선택된 항목은 강조 표시되며 다른 항목을 클릭하면 패널 내용만 갱신된다). **[구현 시 확인] 이 패널이 실제로 보여주는 필드는 색상 점·유형 배지·Confidential 배지·이름·고객사·기간·Description·해시태그·참여 인력(직급순, 각자의 수행 기간)·Engagement No.이며, Open/Closed 상태 배지는 원본 컴포넌트(Engagement 검색)에도 애초에 없다** — "그대로 재사용"이 원칙이므로 CV Generator만을 위해 상태 배지를 새로 추가하지 않았다(추가하려면 Engagement 검색 쪽에도 동일하게 적용해야 특정 화면만 컴포넌트가 갈라지는 것을 피할 수 있음). **Confidential 필드 마스킹(W-7)**은 이 패널에서도 동일하게 적용된다(viewer·assistant는 이름·고객사·Description·해시태그만 "(Confidential)", 기간·참여 인력은 노출 — 서버 `work_items_safe`가 이미 마스킹한 값을 그대로 표시할 뿐이므로 별도 처리 불요). V-5의 검색/필터로 좁혀진 목록에서도 항목 클릭 시 동일하게 상세 패널이 열린다.

- **드릴다운(D-6)**: 대시보드의 목록 항목은 클릭 시 해당 화면으로 이동한다. ① 업무지정 필요대상·복귀 대상의 **이름 클릭 → 타임라인의 해당 인력 행으로 이동(스크롤·하이라이트)**. **세로 스크롤 버그 수정(v2.95)**: 노란색 하이라이트는 되지만 세로 스크롤이 대상 인력 행까지 내려가지 않던 문제 — 계산된 픽셀 오프셋(`rowTops`) 기반 스크롤은 레인 패킹 결과·타이밍에 따라 실제 브라우저 레이아웃과 어긋날 수 있었다. 각 행 DOM에 `data-row-key` 식별자를 부여하고, 행 렌더가 끝난 뒤 대상 인력 행을 실제 DOM에서 찾아 `scrollIntoView({block:'center'})`로 스크롤하도록 변경 — 계산 오차에 영향받지 않는다. 가로(T-16의 focus=오늘)와 세로(대상 인력) 스크롤은 서로 독립적으로 호출해 하나가 다른 하나를 덮어쓰지 않는다. ② Kick-off·종료 목록의 **프로젝트 클릭 → 작업항목 상세 표시**(위 3건 수정 적용). viewer에게도 열람 범위 내에서 동일 동작(마스킹 유지).

| G-3 | 결과 선택 시 해당 위치로 점프: 인력 → 타임라인 해당 행(하이라이트), 작업항목 → 작업 상세 페이지. |

| **G-5** | **검색어 AND/OR 문법(전 검색 기능 공통)**: 전역 검색(G-1~G4)뿐 아니라 **Engagement 검색(§5.8), CV Generator 이름 검색·CV 내 Engagement 필터(§5.9 V-5), Leave 탭 인력 검색(LV-2), 연차 관리 탭 인력 검색(AL-2a), Workitem 목록 필터(W-3) 등 앱 내 모든 검색 입력**에 동일한 문법을 지원한다. 검색어를 공백으로 구분해 여러 개 입력하면 기본적으로 **AND**(모두 포함하는 항목만)로 처리하고, 검색어 사이에 대문자 **OR**을 넣으면 그 앞뒤 조건을 **OR**(둘 중 하나 이상 포함)로 처리한다(예: `삼성 OR LG 전략` → `(삼성 OR LG) AND 전략`, 왼쪽에서 오른쪽으로 순차 처리, 괄호는 지원하지 않음). **이 검색어 파서는 단일 공용 유틸리티로 구현하여 모든 검색 UI가 재사용**한다(화면마다 검색 로직을 중복 구현하지 않는다). |

### 5.13 연차 관리 (Annual Leave, **admin 편집 전용 · assistant 열람 전용 탭(v2.100 — editor는 접근 불가로 변경)**)

전략팀 자체 휴가 제도(§5.10 Leave)는 실제 휴가 운영의 기준이다. 연차 관리 탭은 그와 별개로 **회사의 공식 휴가 제도(법정연차·신입사원 휴가) 및 회사 타임시트와의 정합**을 위해 존재하며, 목적은 두 가지다: **(A) 퇴사 정산**(UI 탭 명칭은 "연차 정산", AL-6 참조), **(B) 타임시트 입력 판단을 위한 수치 안내**. *(구 (C) "주간 타임시트 입력 지침 자동 생성"은 §5.14 타임시트 지침 탭으로 분리됨)*

| ID | 요구사항 |
|---|---|
| AL-1 | **viewer·editor 접근 불가**(메뉴 비노출 + 서버 RLS 차단, **editor는 v2.100부터 제외**). **admin은 열람·편집, assistant는 열람만 가능**(메뉴 노출, **수동 보정(AL-2d) 입력** 등 모든 편집 컨트롤 비활성, 서버 RLS도 SELECT만 허용). |
| AL-2 | **법정연차 실시간 계산(저장·배치 폐지, 근로기준법 제60조 + 회사 규정)**: **회사 규정상 연차유급휴가는 매년 7월 1일(회사 회계연도 시작일, §8과 동일 기준일)에 발생**한다. hire_date를 근거로 **조회 시점마다 실시간으로 계산**한다(계산 규칙은 AL-2b). **Partner 직급은 이 자동 계산 대상에서 제외한다(P-3).** **어차피 퇴사 정산(AL-3a)에서도 동일 계산을 실시간으로 다시 수행하므로, 별도의 "적립 관리" 화면에서 연도별 적립 행을 미리 생성·저장·수동으로 추가/수정/삭제하는 기능은 두지 않는다** — 자동 계산 결과는 조회 전용이며 그 자체를 편집할 수 없다. 80% 출근율 미달 등 시스템이 판단할 수 없는 예외에 한해서만 **수동 보정(+/-, 사유 기재, AL-2d)** 을 자동 계산 결과 위에 추가로 얹어 반영한다. |
| AL-2b | **법정연차 실시간 계산 규칙(근로기준법 제60조, 회사 회계연도 7/1 기준)**: ① **입사 연도(1년 미만 근속 기간, UI 라벨: "신입사원 휴가")**: hire_date 기준 매월 개근을 가정해 완료된 개월 수만큼 1일씩, **최대 11일**(근로기준법 §60②) — 월 단위 계산이라 hire_date 자체의 월별 기준으로 누적하며 회계연도 경계와 무관하게 진행된다. **무급휴가 반영은 AL-2e 참조.** ② **입사 후 처음 맞이하는 7월 1일 시점에 아직 만 1년이 안 된 경우(UI 라벨: "법정연차")**: 15일 × (hire_date부터 그 7/1까지의 재직일수 / 365)로 **비례 계산**하며, ①(신입사원 휴가)과 상계·합산하지 않고 **별개 항목**으로 계산한다. **반올림·올림 없이 소수점 그대로(예: 4.4일) 계산·표시한다.** ③ **입사 후 만 1년이 지난 시점부터(UI 라벨: "법정연차")**: **매년 7월 1일**에 **기본 15일** + **근속연수 3년차부터 매 2년마다 1일씩 가산**(가산일수 = `floor((경과 근속연수 − 1) / 2)`, 가산 포함 총 25일 한도). **80% 출근율 미달 시 일할 비례 축소는 AL-2f 참조.** ④ (AL-2f로 이동) ⑤ **이 계산은 어떤 테이블에도 저장하지 않고, 조회할 때마다(AL-6·AL-7 등 필요한 모든 곳에서) 매번 새로 계산하는 순수 함수로 구현**한다. **본 로직은 근로기준법 제60조의 표준 규정 + 회사의 7/1 발생 규정을 코드화한 것이며, 개별 취업규칙·단체협약·노사 합의 등 세부 예외는 반영되어 있지 않으므로, 실제 배포 전 사내 노무 담당자 또는 공인노무사의 검토를 받는 것을 권장한다.** |

**(A) 퇴사 정산 로직**

핵심 원칙: 팀 제도로 정당 적립된 휴가(프로젝트휴가 등)와 법정연차는 **합산되는 별개 권리가 아니라, 둘 중 더 큰 쪽이 총 휴가 권리**가 된다. 단, **주말/휴일대체 휴가**(실제 근무에 대한 법정 성격의 보상)와 **특별휴가**(예비군·경조사 등 연차와 별개로 발생하는 권리)는 법정연차 쪽 후보에도 함께 더해서 비교한다. 즉 구성원은 max(법정연차 누적 + 주말/휴일대체 누적 + 특별휴가 누적, 팀 정당 적립 누적) 이상의 휴가 권리를 가지며, 그 범위 내 사용은 퇴사 시 불이익이 없다. **무급 휴가는 이 비교의 어느 쪽에도 포함하지 않는다. 퇴사 정산에서 법정연차·신입사원 휴가는 아래 AL-3a에 따라 입사일 기준으로만 계산한다(회계연도 기준과 비교하지 않음).**

| **TSG-2** | **주간 지침 생성(3단계 워크플로우 — 초기화 / 미리보기 / 반영). 아키텍처: 일괄·병렬 처리 금지, 인력별 순차 처리로 전면 재설계(v2.66)** — 지금까지 여러 차례 수정에도 불구하고 실행할 때마다 다른 양상으로 결과가 틀리는 것은 로직 자체의 결정론적 오류라기보다 **여러 인력을 한꺼번에 비동기·병렬로 처리하다 생기는 동시성 문제**일 가능성이 높다고 판단, 다음과 같이 처리 방식 자체를 바꾼다. **① 초기화(베이스라인 재설정, 별도 버튼)**: **삭제 범위와 재계산(쓰기) 범위를 구분한다(v2.71 정정 — 이전 버전은 이 둘을 혼동해 문서 자체가 자기모순이었음).** **삭제 범위 = 8주 창 전체(이번 주 포함)**: 초기화를 실행하면 이번 주를 포함해 8주 창 전체에서 기존 스냅샷을 통째로 지운다(예전 "반영"으로 확정해뒀던 이번 주 데이터가 낡은 채로 남아 혼란을 주지 않도록, 이번 주 것도 일단 삭제 대상에 포함). **재계산(쓰기) 범위 = "이번 주(오늘이 속한 주)를 제외한 그 이전 모든 주"만**: 삭제 후 TSG-1로 다시 계산해 스냅샷에 써 넣는 대상은 이번 주를 제외한 과거 주들뿐이다. **따라서 초기화가 끝나면 이번 주는 스냅샷 행이 하나도 없는 완전히 빈 상태여야 한다** — 이번 주 값은 오직 "지침 생성(②)+반영(③)" 사이클을 통해서만 채워진다. 초기화가 이번 주에 어떤 값이든 다시 써 넣으면(과거처럼 재계산해서 채워 넣는 것도, 삭제만 하고 그대로 두는 게 아니라 무언가를 넣는 것도) 버그다 — 초기화 후 이번 주 섹션은 "신규 입력 대상"으로만 보여야 한다. **처리는 인력을 하나씩 순서대로 완전히 끝내가며 진행한다(다음 인력은 이전 인력이 완전히 끝난 뒤에만 시작)**: 재직중이었던 인력 목록을 구한 뒤, 그 목록을 하나씩 순회하며 각 인력 A에 대해 (1) 그 인력의 **8주 창 전체(이번 주 포함) 범위**의 기존 스냅샷 행을 삭제, (2) **현재 타임라인(work_item·assignments·leave 등 라이브 데이터)에 기반해 그 인력의 "이번 주를 제외한" 재계산 대상 범위의 영업일을 TSG-1로 다시 계산**(그 인력이 그 기간 중 퇴사·입사했다면 재직 기간 내의 날짜만 — 아래 ⑨), (3) 계산된 값을 저장(이번 주 날짜에는 어떤 행도 insert하지 않음), (4) 저장 직후 그 인력·그 재계산 범위(이번 주 제외)를 재조회해 기대(그 인력의 재직 기간에 해당하는 영업일 수)와 실제 저장 건수가 일치하는지, 그리고 **이번 주 날짜에는 행이 0개인지** 함께 검증 — **일치해야만 다음 인력으로 진행**하고, 불일치하면 그 인력에서 멈추고 어떤 인력·어떤 날짜가 문제인지 화면에 표시한다(다른 인력에게 넘어가지 않음). 화면에는 "N/전체 — 처리 중: 홍길동" 같은 진행 상황을 실시간으로 보여준다. **이렇게 인력 단위로 삭제→재계산→저장→검증을 완결짓고 넘어가면, 도중에 중단되어도 이미 끝난 인력은 확실히 맞고 아직 처리 안 된 인력은 기존 상태 그대로 남으므로(전체를 먼저 삭제하고 재생성하는 방식보다 안전), 그리고 인력 간 상태 공유가 없으므로 지금까지 반복되던 동시성 관련 버그(같은 사람 일부 요일만 채워짐, 특정 인력만 통째로 누락, 실행마다 다른 결과)가 원천적으로 발생하지 않는다.** ⑥ 이 전체 과정(모든 인력 처리 완료)이 끝나기 전까지 버튼은 비활성·로딩 상태를 유지하고, 사용자가 페이지를 벗어나거나 버튼을 다시 눌러도 새 실행이 겹쳐 시작되지 않도록 막는다. **완료 후 검증**: 초기화 직후 그 범위(이번 주 제외)를 "지침 생성"으로 미리보기하면 반드시 **신규·정정 지시가 0건**이어야 하며, **이번 주는 스냅샷 행이 0개(완전히 비어 신규 입력 대상으로만 표시)** 여야 한다. **⑨ 재직 기간은 날짜 단위로 정확히 반영**: 8주 창 내 퇴사한 인력은 퇴사일 이전까지만, 입사한 인력은 입사일 이후부터만 그 인력의 대상 범위에 포함한다(고정된 "현재 재직중 인력 목록 × 전체 기간" 격자로 계산하지 않는다 — 인력별 순차 처리 구조 자체가 각 인력의 실제 재직 기간을 그 인력 처리 시점에 개별 조회하므로 자연스럽게 보장되어야 한다). **⑩ 특정 7인 공통 데이터 결함 의심(핵심 신규 단서)**: 지금까지 재직 기간 판정 오류가 반복적으로 발생한 인력(황유동, 이준호96 등)과, **타임라인 이름순 정렬(T-4)에서도 정상적으로 정렬되지 않는 인력이 정확히 일치**한다 — **강민주, 박정인, 백종수, 신유진, 이와이, 이준호96, 황유동 7명**. 서로 다른 두 기능(이름 정렬 / 재직기간 기반 타임시트 계산)에서 같은 인력군이 동시에 문제를 일으킨다는 것은, **로직 버그가 아니라 이 7명의 `people`/`profiles` 레코드 자체에 공통된 데이터 결함이 있을 가능성**을 강하게 시사한다. 다음을 직접 조회해 확인한다: (a) 이 7명의 `name` 필드를 다른 정상 인력과 바이트 단위로 비교한다 — 앞뒤 공백, 탭, 줄바꿈, 유니코드 정규화 차이(예: 자모 분리 표현 vs 완성형), 전각/반각 문자 혼입 등이 있는지 확인한다(이름순 정렬이 깨지는 전형적 원인). (b) 이 7명의 `hire_date`/`termination_date` 필드의 실제 저장 형식(문자열 vs 날짜 타입, 타임존 오프셋 포함 여부 등)이 다른 정상 인력과 다른지 확인한다. (c) 이 7명이 **CSV 일괄 업로드(§5.11b)로 한 번에 등록되었는지** 확인하고, 맞다면 그 업로드 배치에 이름·날짜 파싱 관련 공통 버그가 있었는지(예: 업로드 시점의 인코딩 문제, 날짜 형식 오인식 등) 점검한다. (d) 실제 결함을 찾으면 그 7명의 레코드를 정정하고(데이터 정정), 향후 같은 결함이 재발하지 않도록 CSV 업로드 검증(B-3)에 관련 체크(공백 trim, 인코딩 정규화 등)를 추가한다. **⑬ [해결됨] 근본 원인: Supabase 프로젝트의 PostgREST 서버 설정 `Max Rows`가 1000으로 걸려 있었음**: 클라이언트 코드의 `.limit(50000)`은 이 서버 측 상한을 override하지 못해, `timesheet_guideline_snapshot`을 조회하는 모든 쿼리(스냅샷 로드, 지침 생성 내부 재조회 등)가 정렬 기준 없이 1000행에서 잘려 들어왔다. 그 결과 실제로는 1095건이 정상 저장되어 있어도 조회 시마다 임의의 1000건만 도착해 "매번 다른 사람이 빠지는" 것처럼 보였다(황유동·황경윤·황신형, 이태인·이형돈·이준호96·이준호99 등 — 이름과는 무관). **조치**: ① Supabase 대시보드에서 Max Rows 상한 상향, ② 이와 별개로 `timesheet_guideline_snapshot`을 조회하는 모든 지점에 `.range()` 기반 페이지네이션 헬퍼를 적용해 서버 설정값과 무관하게 항상 전체를 가져오도록 코드 레벨에서도 보장(TSG-3④ 참조). 이 조치 이후 1095건 규모에서 정상 동작 확인됨. **⑪ 초기화를 다시 실행하지 않았는데도 탭 전환·창 전환만으로 내용이 바뀌는 버그(신규, 심각)**: 사용자가 "초기화"나 "반영"을 다시 누르지 않았는데도, 단순히 다른 브라우저 탭/창으로 갔다가(alt-tab 등) 돌아오기만 해도 화면의 미달 경고·타임시트 내역이 방금 전과 다르게 표시되는 현상이 보고되었다. **DB에 아무 변경이 없다면 같은 데이터를 다시 조회했을 때 항상 같은 결과가 나와야 하므로**, 이 현상은 다음 중 하나가 원인일 가능성이 높다: (a) **브라우저 탭이 다시 포커스를 받을 때(`visibilitychange`, `focus` 이벤트) 실행되는 코드가 있고, 그 코드가 단순 재조회가 아니라 실제로 무언가를 다시 계산해서 저장하거나, 이전 실행에서 못 끝낸 작업을 이어서 실행하고 있을 가능성** — 관련 이벤트 리스너를 전부 찾아서 어떤 부작용(side effect)이 있는지 확인한다. (b) **이전 초기화/반영 실행이 실제로는 완전히 끝나지 않고 백그라운드에서 계속 진행 중이며(⑥에서 막으려 했던 문제가 여전히 남아있을 가능성), 사용자가 탭을 벗어나 있는 동안에도 계속 실행되다가 돌아왔을 때 마침 그 시점까지 처리된 상태가 보이는 것** — 초기화·반영 함수가 정말로 끝까지 실행이 완료된 후에만 "완료" 상태를 반환하는지, 혹은 함수 호출은 반환됐지만 내부적으로 예약된(setTimeout, 별도 프로미스 체인 등) 후속 작업이 계속 남아있는 건 아닌지 코드를 처음부터 끝까지 추적한다. 있는 드리프트도 함께 발견되어, 각 건을 실제 스키마 상태와 대조 확인한 뒤 `supabase migration repair`로 이력만 정합화하고 나머지는 정상 push했다. **push 후에도 실제 함수 호출은 여전히 실패**했는데, PostgREST 엔드포인트로 직접 호출해 확인한 새로운 원인은 함수 본문의 `RAISE EXCEPTION 'CODE' USING MESSAGE = '...'` 구문 오류(SQLSTATE 42601, "RAISE option already specified: MESSAGE")였다 — PL/pgSQL에서는 RAISE EXCEPTION 뒤의 문자열 리터럴 자체가 이미 MESSAGE를 지정하므로 USING MESSAGE를 추가로 쓰면 충돌한다. admin이 아니거나 인자가 유효하지 않은 모든 호출 경로(즉 정상 admin·정상 인자 경로를 제외한 전부)에서 이 오류가 발생해 의도한 FORBIDDEN/INVALID_* 에러 대신 문법 오류가 노출되고 있었다. 마이그레이션 0035에서 두 함수의 모든 RAISE 문을 단일 포맷 문자열로 통일해 수정했고, PostgREST를 통한 실호출(비인가 호출 → 정상적으로 `FORBIDDEN: ...` 반환)로 검증했다. **⑨ 사용량 검증 오발동(버그 수정, v2.93)**: RPC 자체는 정상 동작했지만, 휴가 블록을 대상으로 "Closed로 잠금" 적용 시 status만 바꾸는 UPDATE인데도 LV-6 특별휴가 사용량 검증이 발동해 실패하는 사례가 있었다. 근본 원인과 수정은 LV-16 참조(검증 트리거에 WHEN 조건 없어 모든 UPDATE에 재발동 + 영업일 계산이 공휴일을 제외하지 않아 Ledger와 잔여가 어긋난 두 가지 문제). |
| **ADM-10** | **초대(invite) 계정의 비밀번호 최초 설정 강제(신설, v2.97 / 정정, v2.98 / 정정, v2.99)**: Supabase Auth의 "Send invitation"으로 생성된 계정은 초대 링크를 클릭해 인증을 마치는 순간 **Supabase가 자체적으로 이미 유효한 로그인 세션을 발급**한다(설계상 정상 동작 — Supabase가 자동으로 비밀번호 설정을 강제해주지 않음). 이 때문에 **URL Configuration을 프로덕션 주소로 바로잡은 뒤에도, 초대받은 사람이 비밀번호를 설정하지 않고 바로 앱에 접속되는 문제**가 남는다. **① [v2.98 근본 원인 확정] "invite 시점에 플래그를 true로 세팅"(v2.97 최초 구현)은 구조적으로 항상 실패한다.** 실제 라이브 데이터로 재현·확인한 원인: Supabase가 `auth.users` 행을 INSERT할 때는 `invited_at` 컬럼이 아직 비어 있고, 그 직후(수 ms 뒤) 별도 UPDATE로 `invited_at`을 채운다. `AFTER INSERT ON auth.users` 트리거에서 `NEW.invited_at IS NOT NULL`을 조건으로 판별하던 v2.97 방식은 이 타이밍 때문에 **초대 경로가 앱이든 Supabase 대시보드든 관계없이 항상 NULL을 보게 되어 플래그가 한 번도 true로 세팅될 수 없었다**(대시보드 직접 초대가 원인이라는 가설은 기각 — 앱을 거친 초대였어도 동일하게 실패했을 조건이었음). **조치(설계 반전 — "신규 계정은 기본 잠금")**: `invited_at` 판별을 완전히 버리고, `auth.users`에 새 행이 생기면(이 프로젝트에서 로그인 계정이 만들어지는 유일한 경로 — 자체 회원가입 없음) 트리거가 **무조건 `must_set_password = true`** 로 `profiles` 행을 생성한다. CSV 일괄 업로드(§5.11b)는 `people`/`work_items` 테이블만 갱신하고 `auth.users`를 전혀 만들지 않음을 코드로 확인했으므로 이 트리거의 영향을 받지 않는다(추가 예외 처리 불요). **② [v2.98] 라우트 가드 fail-closed 정정**: v2.97 구현(`profile?.must_set_password`)은 `profile`이 `null`(조회 실패/미로딩)일 때 옵셔널 체이닝이 `undefined`로 단락되어 **차단 없이 통과(fail-open)** 되는 결함이 있었다. `!profile \|\| profile.must_set_password`로 정정해 **프로필 행을 확인하지 못한 경우도 차단**으로 처리한다(행 없음 = 미검증 = 차단). 로그인된 사용자가 이 조건에 해당하는 동안에는 어떤 경로로 진입하든(직접 URL 이동 포함) 비밀번호 설정 페이지 외의 모든 화면 접근을 차단하고 그 페이지로 강제 리다이렉트한다(대시보드·타임라인 등 어떤 데이터도 먼저 보이지 않아야 한다). 가드는 라우터 최상위 한 곳에만 적용(라우트별 누락 없음, v2.97부터 유지). **③ 비밀번호 설정 페이지**: 새 비밀번호를 입력받아 `supabase.auth.updateUser({password})`로 갱신하고, 성공 시 **`complete_password_setup()` RPC(SECURITY DEFINER)로 그 계정의 `must_set_password`를 false로 갱신**한 뒤에만 나머지 화면 접근을 허용한다(RLS로 일반 UPDATE 경로는 차단되어 있어 이 RPC만이 유일한 해제 경로). RPC 실패 시 에러를 보여주고 재시도 가능. ④ **세션이 이미 발급된 상태라는 점을 전제**로 한다 — 즉 "비밀번호 설정 전에는 로그인이 안 된다"가 아니라 "로그인은 되어 있어도 이 플래그가 꺼지기 전까지는 앱의 실제 기능에 접근할 수 없다"는 방식으로 구현한다(Supabase 세션 자체를 지연 발급할 수는 없으므로). ⑤ 만료·재사용 링크 에러는 사용자 친화적 문구로 안내한다. **⑥ [v2.98 백필]** 기존에 이미 초대됐으나 아직 비밀번호를 설정한 적 없는 계정(초대 링크 클릭 시점의 자동 로그인 1회뿐, 이후 재로그인 이력 없음 — `last_sign_in_at`이 `confirmed_at`으로부터 1분 이내)만 신중하게 골라 일회성으로 `true`로 정정했다. 이미 정상적으로 비밀번호를 설정해 쓰고 있는 admin·기존 계정(재로그인 이력으로 판별)은 이 백필에서 제외되어 영향받지 않는다. **⑦ [v2.99 근본 원인 확정 및 수정] 발급된 임시 비밀번호로 로그인 후 새 비밀번호를 제출하면 진행이 막히는 버그**: 이 페이지는 이미 완전히 독립된 단순 폼이라(현재 비밀번호 입력란을 요구하는 재사용 컴포넌트 문제 아님, 코드로 확인) 원인이 아니었고, 실제로는 `supabase.auth.updateUser({password})` 호출이 **Supabase가 실제로 반환한 에러**(`error_code: "current_password_required"`, `msg: "Current password required when setting new password."`)로 실패하고 있었다 — 콘솔에 원본 에러를 찍어 확인. 이 프로젝트의 Supabase Auth 설정에 **`GOTRUE_SECURITY_UPDATE_PASSWORD_REQUIRE_CURRENT_PASSWORD`가 켜져 있어, 이미 비밀번호가 설정된 계정(admin이 발급한 임시 비밀번호도 포함)의 `updateUser` 호출은 `current_password` 필드를 요구한다**(`@supabase/auth-js`의 `UserAttributes` 타입에 정식 필드로 존재 — 이메일 OTP 재인증이 아니라 비밀번호 문자열 필드 검증 방식임을 확인). 조치: 로그인 페이지에서 사용자가 방금 입력해 검증까지 마친 비밀번호를 `sessionStorage`에 1회용으로 담아 비밀번호 설정 페이지로 넘기고, 그 페이지가 `updateUser({ password, current_password })`로 함께 제출한다(매직링크 등 기존 비밀번호가 아예 없는 계정은 `current_password` 없이 기존과 동일하게 동작). 정상 계정(admin 등, `must_set_password=false`)은 라우트 가드 통과 시점에 이 세션스토리지 값을 즉시 폐기해 평문 비밀번호가 불필요하게 남지 않게 한다. 실제 라이브 프로젝트에 테스트 계정을 만들어 로그인→`current_password` 포함 `updateUser`→RPC→재로그인까지 curl로 전 구간 재현·검증 후 테스트 계정 삭제. **참고**: 이 설정은 `/forgot-password`(비밀번호를 잊어 재설정하는 일반 흐름)의 매직링크 기반 재설정에도 영향을 줄 수 있는 것으로 알려져 있으나, 이번 조사는 ADM-10의 강제 최초설정 흐름에 한정했다 — 비밀번호를 잊은 사용자가 재설정 링크로도 동일한 에러를 만난다면 별도로 확인이 필요하다. |

### 6.1 인증
- Supabase Auth(이메일/비밀번호). 공개가입 비활성·관리자 발급. service-role 키 미노출.

- `profiles.person_id` = `people.id`로만 매칭(관리자 수동 설정). LPN 매칭 미사용. `my_person_id()` = `profiles.person_id`.

| 역할 | 접근 |
|---|---|
| admin | 전체 편집 + 계정·권한·설정·백업 + Admin·Migration |

| **assistant** | *(v2.11)* 지정된 화면(연차 관리·타임시트 지침 포함) **열람 전용**(편집 비활성, RLS로 쓰기 차단). **People·공휴일 동기화는 v2.100부터 비노출**(기존에도 편집 권한은 없었으나 화면 자체를 숨김). **Pipeline 비노출·Confidential 필드 마스킹은 viewer와 동일 적용(W-7, v2.101에서 서버·클라이언트 전 구간 재검증 — 서버는 이미 동일 적용 중이었고 클라이언트 잔여 결함 2건만 수정)**. |
| viewer | editor와 동일 화면 **전면 읽기 전용**. ① 모든 편집 비활성(서버 RLS 차단), ② CV·Leave는 본인(person_id)만, ③ Pipeline 비노출(W-7①), ④ Confidential 필드 단위 마스킹(W-7②, 행은 노출), ⑤ Admin·Migration·People·연차 관리·타임시트 지침 비노출 |

### 6.4 Closed 잠금(전 역할)
- Closed 작업(work_item, §5.5)과 Closed 휴가 블록(assignments kind='leave', LV-15)은 admin·editor에게도 편집 비활성. Open 전환 후 편집. 서버 강제.

## 7. 휴가 산정 규칙
1. 프로젝트휴가: **인력×프로젝트 단위 합산 적립** — 한 인력이 같은 프로젝트에 둘 이상의 분리된 배정으로 투입된 경우, 배정별로 각각 round 적립하지 않고 **해당 프로젝트 내 모든 투입 구간(각각 본 프로젝트 구간과의 교집합)의 영업일수 합**(주말·공휴일 제외, §8과 동일한 영업일 정의)을 먼저 구한 뒤 **round(합계 투입 영업일수/10)** 로 1회 적립한다. Pre-study 제외. (예: 12영업일+9영업일로 나뉜 투입 → round(21/10)=2일. 배정별 계산 round(12/10)+round(9/10)=1+1과 다름에 유의.) **(버그 수정) 이 항목은 이전에 "달력일수"로 잘못 기재되어 있었다 — 주말·공휴일을 포함한 전체 일수가 아니라, 영업일수만으로 계산해야 한다.**
2. **주말/휴일대체**: 실근무일만 인정. **주말(토요일·일요일 모두 포함)** 1일당 **0.5일**, **공휴일** 1일당 **1.0일** 적립(pipeline 제외). **주말과 공휴일이 겹치는 날(예: 공휴일이 토·일요일인 경우)은 주말 기준(0.5일)을 우선 적용**하며, 공휴일 가산을 별도로 중복 적용하지 않는다. **(재발 버그 방지) 이 규칙은 타임라인의 "주말 실근무 마커"(T-7) 토글을 포함해, 주말/휴일대체 적립이 발생하는 모든 진입점(마커 토글, 배치 계산, 수동 적립 등)이 동일한 단일 계산 함수를 호출해 적용해야 한다 — 특정 진입점(예: 실근무 마커)만 별도의 하드코딩된 +1일 로직을 갖고 있어 이 규칙을 우회하는 일이 없어야 한다. 일요일 실근무 마커 체크 시에도 예외 없이 0.5일이 적용되는지 반드시 확인한다.**
3. **지연보상(수동 전환)**: 지연보상은 **자동 산정하지 않는다**(기존 자동 판정 로직 제거). 프로젝트 종료 후 휴가 사용 지연이 자의인지 회사 사정인지는 시스템이 판단할 수 없으므로, **관리자가 휴가 패널의 수동 적립으로 '지연보상' 유형을 직접 부여**한다(권장 기준 참고치: 적립분 ≤1:0 / 1.5~3:1 / 3.5~5:2 / 5.5↑:3 — UI 도움말로만 안내). |
4. 사용·잔여 총량 정합(FIFO 원천 표시용, 음수 허용). 수동 +/-.
5. 자동 잔여 배정: §5.10.
6. 휴가 영업일 스냅.
7. **소급 재계산(v2.13 버그 수정 적용)**: 위 규칙2 변경 시점 이전에 "주말+공휴일 겹침일을 공휴일(1.0일)로 과다 적립"했거나 "일요일을 주말로 인식하지 못해 0.5일을 누락 적립"한 기존 accruals 이력을 **일회성 마이그레이션으로 자동 재계산·정정**한다. 별도 관리 화면·미리보기·확인 UI 없이, 배포 시 한 번 실행되는 스크립트가 대상 건을 찾아 규칙2 기준 올바른 값과의 차액만큼 보정(+/-) 항목을 자동으로 적립한다(기존 행은 삭제하지 않고 보존). 사유는 "v2.13 주말/공휴일 겹침 산정 버그 소급 정정"으로 기재하고 audit_log에 기록한다.

### 9.1 색상/표기/레이아웃
- 색상 파생(§4). MM/DD. 반응형. Admin·Migration admin 전용. Undo/Redo 버튼. 호버 툴팁, 중복 배정 경고, sticky 헤더/라벨, 즉시 필터+필터칩, 모달 Esc/Enter, 빈 상태 안내. viewer 편집 컨트롤 전면 비활성.
- **와이드 스크린 레이아웃 활용**: Leave 탭(Leave Ledger)과 연차 관리 탭의 퇴사 정산 화면처럼 상세 내역(적립/사용 리스트, FY 소계 등)이 길게 나열되는 화면은, 좁은 화면 폭에 고정(예: 중앙 정렬된 좁은 max-width 컨테이너)되지 않고 **화면이 넓을 때는 그 너비를 활용**하도록 반응형으로 확장한다. 특히 AL-6의 "회계연도 기준"과 "입사일 기준" 두 계산 내역처럼 나란히 비교할 항목이 있는 경우, 와이드 스크린에서는 **2열 나란히 배치**를 우선 고려하고(좁은 화면에서는 기존처럼 위아래로 스택), 리스트 자체도 폭이 넓어진 만큼 컬럼(날짜·유형·원천·일수 등)이 잘리지 않고 여유 있게 표시되도록 한다.
- **날짜 입력 형식 통일(YYYY-MM-DD, 버그 수정)**: 일정·배정·인력·휴가 등 **날짜를 직접 입력받는 모든 필드**(달력 위젯이 아닌 텍스트 입력, 또는 달력 위젯이라도 텍스트로 타이핑 가능한 경우 포함 — 인력 관리의 입사일/퇴사일, Workitem의 시작·종료일, 휴가 배정 기간, 연차 관리의 기준일, CSV 업로드 템플릿의 날짜 컬럼, 공휴일 관리, 그 외 앱 전체의 모든 날짜 입력)은 **MM-DD-YYYY 형식이 아니라 YYYY-MM-DD 형식으로 입력받도록 통일**한다. 이는 화면에 막대·라벨을 표시할 때 쓰는 MM/DD 표기(T-1, 타임라인 시각화용 축약 표시)와는 별개다 — **입력 필드는 항상 YYYY-MM-DD, 타임라인 등 시각화 표시는 기존처럼 MM/DD를 유지**한다.

### 9.2 좌측 네비게이션 탭 순서
- 좌측 탭의 노출 순서는 다음과 같이 고정한다: **① 대시보드 → ② 타임라인 → ③ Work Items → ④ Engagement 검색 → ⑤ CV Generator → ⑥ People(인력 관리) → ⑦ Leave(휴가 패널) → ⑧ 연차 관리 → ⑨ 타임시트 지침(§5.14) → ⑩ Admin**. 역할별로 일부 탭이 보이지 않더라도(예: viewer는 People·Admin·연차관리·타임시트 지침 비노출, **editor는 v2.100부터 People·연차관리·타임시트 지침·Admin 비노출**, assistant는 People·Admin 비노출) 노출되는 탭들 사이의 상대적 순서는 이 순서를 그대로 따른다.
