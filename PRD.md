## 1. 개요

전략팀 대시보드는 프로젝트 기반 조직의 인력·일정·휴가·가동률을 통합 관리한다. 간트 타임라인, 휴가 자동/수동 산정, Utilization 대시보드, 개인 CV, Engagement 검색을 제공하며 권한을 RLS로 강제한다.

## 2. 역할

| 역할 | 접근 |
|---|---|
| admin | 전체 편집 + 계정·권한·설정·백업 + Admin·Migration 메뉴(admin 전용) |
| editor | 전체 데이터 열람·편집 |
| viewer | editor와 동일 화면을 보되 **전면 읽기 전용**. CV Generator·Leave는 **본인 정보만**, **Pipeline 비노출**, **Confidential 마스킹**, Admin·Migration 비노출 (§6.3) |
| assistant | **viewer와 모든 면에서 동일**(전면 읽기 전용·Pipeline 비노출·Confidential 마스킹·Admin/Migration 비노출). **단 CV Generator·Leave를 본인만이 아니라 전원(모든 인력) 열람 가능**한 점만 다름 (§6.3) |

## 3. 데이터 모델 (요지)

| 테이블 | 주요 컬럼 |
|---|---|
| profiles | id, name, global_role, person_id(→people.id, 매칭키), lpn, status |
| people | id, lpn(unique), name, rank, role, hire_date, termination_date |
| work_items | id, type(project/proposal/pipeline), name, start, main_start, end_date, status(open/closed) — 전 유형, engagement_number, client, hashtags[], description, confidential(bool) (color 미사용, 유형 파생) |
| assignments | id, person_id, kind(work/leave), work_item_id, weekend_dates[], leave_type, start, end_date, note |
| accruals | id, person_id, direction(accrual/usage), type, days(+/-), date, source, note |
| leave_types | name, active(bool), sort_order |
| holidays | id, name, date, recurring, source('auto'/'manual', 기본 'manual') |
| settings | key, value (fiscal_year_start_month=7) |
| audit_log | id, user_id, action, target_type, target_id, at |

> 입력 형식: Engagement Code `E-00000000`(E- + 숫자 8자리), LPN 숫자 5자리 `00000`. 계정↔인력 매칭은 `profiles.person_id`=`people.id`로만(LPN 미사용).

## 4. 색상 체계
- project=파랑, proposal=노랑, pipeline=**빨강**, Leave=녹색. 각 계열 내 자동 음영. 수동 지정 없음. 유형 변경 시 색 자동 전환.

## 5. 기능 요구사항

### 5.1 실행취소 / 재실행
- Ctrl+Z=실행취소, Ctrl+Y(및 Ctrl+Shift+Z)=재실행. 최대 10단계. 대상: 배정·작업항목·인력·휴가 수동조정. 서버 역연산, 불가 시 안전 실패. editor/admin만. Undo/Redo 버튼.

### 5.2 타임라인 시각화
| ID | 요구사항 |
|---|---|
| T-1 | 인력별/작업별 뷰. 월→주→일 자동 전환. 모든 날짜 MM/DD. 반응형. 막대 라벨 폭 확장 + 호버 툴팁(작업명·고객사·기간, 인력별 뷰는 인력명). **초기 디폴트 배율은 화면에 약 3개월(±1.5개월, 기준=오늘)이 들어오도록 설정**한다(첫 진입·초기화 시 pxPerDay를 뷰포트 폭÷약 90일 기준으로 산정). 사용자가 확대/축소·이동하면 그 상태를 유지하되, 초기값과 '오늘로 이동/리셋' 동작 시 3개월 배율로 복귀. |
| T-2 | 색상: project 파랑 / proposal 노랑 / pipeline **빨강** / Leave 녹색. 범례 구분. |
| T-3 | 작업 행은 그 작업 자체의 일정(span)만 표시(인원 막대 미중첩). 인원별 일정은 드롭다운으로. |
| T-4 | 작업 행·'휴가(전체)' 행 드롭다운 펼침/접힘. 작업 내 인원 직급순·이름순. |
| T-5 | 작업항목 더블클릭 → 상세 페이지(§5.7). |
| T-6 | Pre-study 구간 별도 분류·시각 구분(적립·CV 수행기간 제외). |
| T-7 | 인력 칩 드래그 배정(직급 그룹·이름순). 주말·공휴일 음영, 오늘 세로선/버튼, 주말 실근무 마커 날짜 고정, 정렬·필터·FY/기간 필터, 중복 배정 경고, sticky 헤더/라벨. |
| **T-8** | **헤더·본문 스크롤 동기화(버그 수정)**: 날짜 캘린더 헤더(상단)와 작업/인력 타임라인 본문(하단)이 **항상 동일한 가로 오프셋으로 함께 스크롤**되어야 한다. 특히 FY 필터로 조회 범위가 바뀌거나 좌우로 스크롤할 때 상·하단이 따로 움직여 날짜와 막대가 어긋나는 문제를 해결한다(공통 스크롤 컨테이너 또는 스크롤 위치 동기화). |
| **T-9** | **FY 다중 걸침 처리**: 한 작업/배정의 기간이 둘 이상의 FY에 걸쳐 있어도, FY 필터·좌우 스크롤 시 헤더 날짜와 막대 위치가 어긋나지 않게 한다(좌표 계산 기준을 단일 viewport 원점으로 통일). |
| **T-11** | **Timeline-Person 뷰에 인력 칩 팔레트 표시(버그 수정)**: 현재 Work Item 뷰에만 있는 **드래그용 인력 칩 목록(person chip palette, 직급 그룹·이름순)** 을 Person 뷰에도 **동일하게 표시**한다. 이 칩 목록은 타임라인 옆/위에 위치하며 끌어다 놓아 배정을 생성하는 용도다. Work Item 뷰의 팔레트 컴포넌트를 Person 뷰에서도 재사용(공통 컴포넌트화)하여 두 뷰에서 같은 칩 목록이 보이도록 한다. (T-7의 인력 칩 드래그 배정과 동일 요소) **단, 인력 칩 팔레트는 editor/admin에게만 표시하고 viewer에게는 노출하지 않는다**(viewer는 드래그 배정 불가·읽기 전용이므로 두 뷰 모두 칩 팔레트 비표시). |
| **T-12** | **인력 칩 형광 하이라이트(클라이언트 전용, 비저장)**: T-11의 인력 칩 목록에서 **칩을 클릭하면 해당 칩이 형광색(밝은 노랑 `#FFFF00` 계열)으로 하이라이트** 된다. **복수의 인력을 동시에 선택**할 수 있다. 선택 상태는 Person 뷰와 Work Item 뷰의 칩 목록에서 **공유·동기화**되어, 한 뷰에서 형광 표시한 인력이 다른 뷰의 칩 목록에서도 형광으로 보여 드래그 배정 시 어떤 인력을 배정할지 참고할 수 있다. 하이라이트 상태는 **클라이언트 인메모리(React 상태, `Set<personId>`)로만 관리**하며 DB 저장·타 접속자 공유를 하지 않는다(새로고침 시 초기화). 칩 재클릭 시 해당 인력만 해제, 별도 '선택 해제' 동작으로 전체 해제. (칩 팔레트 자체가 viewer 비노출이므로 본 기능도 editor/admin 전용) |
| **T-13** | **Timeline-Person 이름 검색**: Person 뷰에 **인력 이름 검색 입력**을 두어, 입력어로 인력을 필터링해 해당 인력 행만 표시한다(부분 일치, 대소문자 무시). 검색창은 **Work Item 뷰에서 Client명 검색이 정렬 컨트롤과 같은 줄(라인)에 배치된 것과 동일하게**, Person 뷰의 **정렬(sort) 컨트롤과 같은 줄에 나란히 배치**한다. 검색 중에도 타임라인 헤더·스크롤 동기화(T-8)는 유지된다. 입력 비우면 전체 인력 복원. (CV Generator·Leave의 이름 검색과 동일한 UX) |
| **T-14** | **왼쪽 인력 목록 더블클릭 → 칩 하이라이트 연동**: Person 뷰 **좌측 인력 목록(행 라벨)의 인력을 더블클릭하면**, 상단 인력 칩 팔레트(T-11)에서 해당 인력의 칩이 **형광 하이라이트(T-12)** 된다. T-12와 **동일한 선택 상태(`Set<personId>`)를 공유**하므로 복수 인력 선택·양 뷰 동기화가 그대로 적용된다. 더블클릭은 칩 클릭과 동일하게 **토글**(이미 선택된 인력 재더블클릭 시 해제)로 동작한다. editor/admin 전용(칩 팔레트가 viewer 비노출이므로). |
| **T-15** | **인력별(Person) 뷰 휴가 막대 미표시 버그 수정**: Person 뷰의 각 인력 행에 **work 배정뿐 아니라 휴가(`kind='leave'`) 배정도 막대로 표시**되어야 한다(현재 휴가가 보이지 않는 버그). 휴가 막대는 **Leave=녹색(§4, leave_type별 음영)** 으로 렌더링하고, 호버 툴팁에 휴가 유형·기간을 표시한다. 원인 점검: ① 데이터 조회 시 `assignments`를 `kind` 구분 없이(work+leave 모두) 가져오는지, ② 렌더 분기에서 `kind='work'`만 그리고 leave를 누락하는지, ③ 색상/좌표 계산이 work 전용으로 한정돼 leave가 안 그려지는지 확인. work·leave를 동일 좌표 로직(viewportStart 원점 + pxPerDay)으로 함께 렌더한다. |
| **T-16** | **파트너 복수 동시 배정 멀티 레인 표시(파트너 전용)**: **Partner 직급만** 동시에 여러 work item에 병행 배정될 수 있다(겹침 허용). Person 뷰에서 파트너 행에 기간이 겹치는 배정이 생기면 **레인(lane)을 늘려 여러 줄로 분리**해 표시한다. work item마다 줄을 만들지 않고 **조회 결과 기준 최소 레인 수**가 되도록 그리디 레인 패킹(시작일 정렬 후 마지막 막대 종료일 < 새 막대 시작일인 레인에 우선 배치, 없으면 새 레인 생성). 겹치지 않는 배정은 같은 레인 재사용. 파트너 행 높이는 레인 수에 따라 가변. **드래그 이동/생성 중 새 겹침이 발생하면 레인 패킹을 즉시(실시간) 재계산해 드래그 도중에도 레인이 늘어나야 한다**(현재 드래그 시 레인이 실시간 반영되지 않는 버그 수정 — 드롭 이후에만 갱신되거나 아예 갱신되지 않는 문제). 레인 재계산은 드래그 미리보기 상태(임시 좌표 포함)를 입력으로 매 프레임/이동 이벤트마다 수행한다. **비(非)파트너는 겹침 자체를 허용하지 않으므로 레인이 늘지 않으며 E-6의 클램프/푸시 규칙을 따른다.** |
| **T-16a** | **파트너 휴가끼리는 겹침 금지**: 파트너의 겹침 허용은 **work 배정(`kind='work'`)에 한한다.** **휴가(`kind='leave'`)는 파트너라도 서로 겹칠 수 없다** — 휴가 vs 휴가는 비파트너와 동일하게 E-6의 비파트너 규칙(겹침 금지·푸시/클램프)을 적용한다. (work vs leave, work vs work는 파트너 한정 겹침 허용·레인 분리.) 즉 같은 파트너의 **두 휴가 일정이 동시에 존재할 수 없다.** |

### 5.3 일정 편집
| ID | 요구사항 |
|---|---|
| E-1 | 빈 칸 드래그/더블클릭/칩 드롭으로 생성. 인력별 뷰 빈 칸 클릭 시 해당 인력 자동 선택. |
| E-2 | 작업 선택: status=open 만(전 유형). 인력: 재직 인원만. 휴가 유형: active 만. |
| E-3 | 휴가 영업일 스냅. 드래그 겹침 방지 푸시(같은 인력 비중첩). 푸시 방향 규칙은 E-6. |
| E-4 | viewer는 모든 편집 컨트롤 비활성(서버 RLS로도 차단). |
| **E-5** | **모바일(너비 768px 미만)에서는 역할과 무관하게 모든 편집 컨트롤 비활성.** 빈 칸 클릭·드래그·드롭으로 인한 생성/이동 입력이 동작하지 않아야 한다(§6.6). |
| **E-6** | **겹침 처리 — 파트너/비파트너 구분(버그 수정)**: <br>**(가) 비(非)파트너**: 같은 인력의 일정은 **절대 겹칠 수 없다.** 인접한 두 블록 A→B(A 앞, B 뒤)에서 — ① **A의 종료일을 뒤(미래)로 늘리거나 A를 뒤로 이동**해 B와 겹치면 **B를 미래 방향으로 밀어낸다**(downstream push, 연쇄 적용). ② **B의 시작일을 앞(과거)으로 당기는 경우 A는 밀리지 않으며**, **B의 시작일은 A의 종료일보다 과거로 갈 수 없고 A 종료일 다음 가용일(영업일 스냅)에서 클램프되어 멈춘다.** **현재 B를 과거로 당길 때 클램프가 동작하지 않아 겹침이 생기는 버그를 수정**한다. 즉 푸시는 미래 방향으로만, 과거 방향은 클램프. <br>**(나) 파트너**: **work 배정끼리(work vs work, work vs leave)는 겹침 허용** → 푸시·클램프 없이 레인을 늘려 표시(T-16). **단 휴가끼리(leave vs leave)는 파트너라도 겹침 금지** → (가)의 비파트너 규칙 적용(T-16a). <br>※ 확대/축소·이동 등 좌표 변환 중에도 (가)·(나) 규칙이 동일 적용된다. 드래그 종료(드롭) 시점뿐 아니라 드래그 중 실시간 미리보기에서도 경계를 넘지 않아야 한다. |
| **E-7** | **짧은 블록(1~2일) 이동/리사이즈 영역 분리(버그 수정)**: 현재 1~2일짜리 짧은 블록은 좌우 **리사이즈 핸들이 블록 전체 폭을 점유**하여, 커서가 이동(손 모양)으로 보여도 실제로는 기간 축소·확대가 되는 문제가 있다. 수정: ① 블록 폭이 좁아 이동 영역을 확보할 수 없을 때는 **리사이즈 핸들의 hit 영역을 가장자리 소폭(예: 좌우 각 ≤4px 또는 블록폭의 일정 비율)으로 제한**하고 **중앙은 이동(move) 영역으로 보장**한다. ② 그래도 폭이 너무 좁으면(예: 임계폭 미만) **리사이즈 핸들을 비활성화하고 전체를 이동 영역으로** 처리하며, 기간 조정은 더블클릭/편집 모달 등 대체 수단으로 제공한다. ③ **커서 모양과 실제 동작을 일치**시킨다(손 모양=이동, 좌우 화살표=리사이즈). 리사이즈/이동 판정은 포인터 다운 위치가 핸들 hit 영역 안인지로 결정한다. |

| **LV-5** | **기준일 기준 적립·사용 6분류 계산·표기**: 휴가 적립/사용/잔여를 **기준일(as-of date, 기본=오늘)** 을 기준으로 아래 6개 값으로 구분해 계산·표시한다.<br>① **실제 적립(actual accrued)**: **프로젝트가 완료된(end_date ≤ 기준일, 즉 종료일이 지난) 프로젝트**에서 발생한 적립분만 합산. FIFO 원천·총량 정합은 LV-1과 일관. |
| **LV-6** | **행 클릭으로 상세 열기**: Leave 목록에서 **별도 상세 버튼 없이 해당 행(인력) 전체를 클릭하면 휴가 상세 페이지/패널이 열린다.** 행 전체가 클릭 가능 영역(커서 pointer). viewer는 본인 행만, assistant·editor·admin은 전원 행 클릭 가능. (LV-3의 강제 모달/자동 재오픈 금지 규칙과 충돌하지 않게, 클릭 시에만 인라인 상세 표시) |

| HOL-3 | **동기화 범위·주기**: API가 **요청당 1개월 단위**, 기관 수기입력분이 **대략 향후 1년치**만 제공되는 한계가 있으므로, 동기화는 **현재 연도와 차기 연도(또는 향후 12~18개월)를 월별로 순회 호출**해 가져온다. 주기적 실행(예: 일/주 단위 스케줄 또는 admin 수동 '지금 동기화' 버튼)으로 임시공휴일·선거일 신규 지정분을 반영한다. |
| HOL-4 | **수동 보정 우선 유지(머지 정책)**: admin이 수동으로 추가/수정한 공휴일은 자동 동기화가 덮어쓰지 않도록 출처 구분(`source`: 'auto' / 'manual')을 두고, **manual 항목은 보존**, auto 항목만 갱신. 동기화 결과(추가/변경 건수)는 audit_log 기록. |
| HOL-5 | **admin 전용 UI**: Admin 메뉴에 '공휴일 동기화' 화면(마지막 동기화 시각, 지금 동기화 버튼, 동기화 로그, 수동 추가/편집). 동기화 실패(API 오류/쿼터 초과) 시 안내·재시도. |

## 6. 인증·권한

### 6.1 인증
- Supabase Auth(이메일/비밀번호). 공개가입 비활성·관리자 발급. service-role 키 미노출.

### 6.2 계정↔인력 매칭
- `profiles.person_id` = `people.id`로만 매칭(관리자 수동 설정). LPN 매칭 미사용. `my_person_id()` = `profiles.person_id`.

### 6.3 역할별 접근
| 역할 | 접근 |
|---|---|
| admin | 전체 편집 + 계정·권한·설정·백업 + Admin·Migration |
| editor | 전체 데이터 열람·편집 |
| viewer | editor와 동일 화면 **전면 읽기 전용**. ① 모든 편집 비활성(서버 RLS 차단), ② CV·Leave는 본인(person_id)만, ③ Pipeline 비노출, ④ Confidential 마스킹, ⑤ Admin·Migration 비노출 |
| assistant | **viewer와 동일**(①③④⑤ 모두 적용, 전면 읽기 전용). **②만 완화: CV·Leave를 전원(모든 인력) 열람 가능.** 즉 읽기 전용·Pipeline 비노출·Confidential 마스킹·Admin/Migration 비노출은 viewer와 같고, CV Generator와 Leave 패널에서 본인뿐 아니라 모든 인력의 정보를 볼 수 있다. 편집은 여전히 불가. |

### 6.4 Closed 잠금(전 역할)
- Closed 작업은 admin·editor에게도 **일반 편집(필드 수정·배정 추가/삭제) 비활성**. 단 **Open/Closed toggle 컨트롤은 editor/admin에게 항상 활성** 유지되어 toggle로 Open 복귀 가능(§5.5 W-4·W-5). viewer는 toggle도 비활성. 서버 트리거는 status 변경은 허용하되 Closed 상태에서의 필드·배정 쓰기를 거부(§5.5 W-6).

### 6.5 RLS 요지
- people: 인증 전 역할 SELECT 전체(쓰기 editor/admin). work_items: viewer·assistant는 type≠pipeline만(마스킹 work_items_safe). assignments: viewer·assistant는 pipeline 연결분 제외. accruals: **viewer는 본인만, assistant는 전원 SELECT 가능**(쓰기는 불가). 전 테이블 쓰기 editor/admin + Closed 잠금. (부록 B)
- **assistant 역할**: viewer와 동일 정책에서 **accruals(휴가)·CV 관련 SELECT만 전원 허용**으로 확장. 쓰기·편집은 viewer와 동일하게 전면 차단.

### 6.6 모바일 읽기 전용 (신규)
| ID | 요구사항 |
|---|---|
| MOB-1 | **화면 너비 768px 미만은 모바일로 판단**한다. `useMobile()` 훅으로 `window.innerWidth`를 감지하며, resize 이벤트도 구독해 창 크기 변경 시 실시간 반영한다. |
| MOB-2 | 모바일에서는 **계정 역할(admin/editor/viewer)과 무관하게** 모든 편집·생성·삭제·상태변경 컨트롤이 **비활성(hidden 또는 disabled)** 처리된다. 데이터 조회·열람은 정상 동작한다. |
| MOB-3 | 모바일 접속 시 화면 상단에 **"모바일에서는 읽기 전용으로 제공됩니다"** 안내 배너를 표시한다. |

## 7. 휴가 산정 규칙
1. 프로젝트휴가: (배정 ∩ 본 프로젝트 구간) 달력일수 → round(투입일수/10). Pre-study 제외.
2. 주말/휴일대체: 실근무일만 **주말(토·일) 0.5·공휴일 1.0** (pipeline 제외). **일요일은 주말이므로 0.5일이 적립되어야 한다**(공휴일로 오분류해 1.0일 적립되는 버그 수정). 분류 우선순위: **해당 날짜가 토요일·일요일이면 무조건 주말(0.5)로 판정**하고, 토·일이 아닌 평일 중 공휴일 캘린더에 해당하는 날만 공휴일(1.0)로 판정한다. (주말과 공휴일이 겹쳐도 주말 0.5 우선)
3. 지연보상: 종료 후 지정휴가 아닌데 15일+ 지연 시 ≤1:0/1.5~3:1/3.5~5:2/5.5↑:3.
4. 사용·잔여 총량 정합(FIFO 원천 표시용, 음수 허용). 수동 +/-.
5. 자동 잔여 배정: §5.10 LV-1. 기준 시작점은 **현재/가장 최근 일정의 종료일 다음 영업일**(과거 빈칸이 아니라 마지막 일정 뒤).
6. 휴가 영업일 스냅.
7. **Partner 제외**: Partner 직급은 **프로젝트 투입에 따른 자동 휴가 생성 규칙(1 프로젝트휴가, 2 주말/휴일대체, 3 지연보상)을 적용하지 않는다.** 파트너에게는 투입 기반 적립이 발생하지 않으며, 자동 잔여 배정(§5.10) 대상에서도 제외한다. (필요 시 수동 +/- 조정만 허용)
8. **기준일 기준 실제/예정 분해(LV-5)**: 적립은 **프로젝트 완료(end_date ≤ 기준일) 시점에 '실제 적립'으로 전환**되고, 완료 전에는 '적립 예정'으로만 집계한다. 사용은 **데일리**로, 기준일까지 경과한 휴가일은 '실제 사용', 미래분은 '사용 예정'으로 분해한다. 현재 잔여=실제적립−실제사용, 잔여 예정=(실제적립+적립예정)−(실제사용+사용예정).

### B.3 viewer / assistant RLS / 마스킹 / Closed
- people SELECT: 인증 전 역할 / 쓰기 editor·admin.
- work_items SELECT: editor·admin 전체, **viewer·assistant는 type<>'pipeline'**. 읽기는 work_items_safe(기밀 마스킹) 경유.
- assignments SELECT: **viewer·assistant는 pipeline 연결분 제외**.
- accruals SELECT: **viewer는 person_id=my_person_id()**, **assistant는 전원(제한 없이 SELECT) 허용**. 쓰기는 둘 다 불가(editor·admin만).
- CV 관련 조회: viewer는 본인만, **assistant는 전원** 열람 허용(읽기 전용).
- 전 테이블 쓰기 editor·admin. assistant는 viewer와 동일하게 쓰기 전면 차단. Closed 트리거: work_items UPDATE 시 **status(open↔closed) 변경은 허용**, status 외 컬럼 변경은 거부 + Closed 작업 연결 assignments 쓰기 거부(§5.5 W-6).
- **대량 업로드 RPC는 admin 전용**: work_items 일괄 교체/추가 RPC는 my_role()='admin'일 때만 실행, 그 외 거부. 전체 교체·추가 모두 단일 트랜잭션으로 처리(§5.5a BU-1·BU-5).

### 부록 C. 작업항목 대량 업로드 양식 (Bulk Upload Template)

CSV(UTF-8) 또는 XLSX. **1행=헤더(아래 컬럼명 고정)**, 2행부터 데이터.

| 컬럼명 | 필수 | 설명/형식 |
|---|---|---|
| `type` | 필수 | project / proposal / pipeline 중 하나 |
| `name` | 필수 | 작업항목 이름 |
| `engagement_number` | 선택 | Main Engagement Code. 형식 `E-00000000`(E- + 숫자 8자리) |
| `client` | 선택 | 고객사명 |
| `start` | 선택 | 전체 시작일 `YYYY-MM-DD` |
| `main_start` | 선택 | 본 프로젝트 시작일 `YYYY-MM-DD`. (start ≤ main_start) |
| `end_date` | 선택 | 종료일 `YYYY-MM-DD`. (main_start ≤ end_date) |
| `status` | 선택 | open / closed (기본 open) |
| `description` | 선택 | 설명 |
| `hashtags` | 선택 | 해시태그. `;` 또는 `,` 구분 다중값 (예: `금융;전략`) |
| `confidential` | 선택 | true / false (기본 false) |

검증 규칙: ① 필수 컬럼·값 존재, ② type 유효값, ③ engagement_number 형식 일치(있을 때), ④ 날짜 형식 및 start ≤ main_start ≤ end_date(존재하는 값끼리), ⑤ status·confidential 파싱. 오류 행은 커밋 차단(BU-4). 모드(Replace/Append)는 화면 선택(BU-3).
