| 항목 | 내용 |

## 1. 개요

전략팀 대시보드는 프로젝트 기반 조직의 인력·일정·휴가·가동률을 통합 관리한다. 간트 타임라인, 휴가 자동/수동 산정, Utilization 대시보드, 개인 CV, Engagement 검색을 제공하며 권한을 RLS로 강제한다.

## 2. 역할

| 역할 | 접근 |
|---|---|
| admin | 전체 편집 + 계정·권한·설정·백업 + Admin·Migration 메뉴(admin 전용) |
| editor | 전체 데이터 열람·편집 |
| **assistant** | *(v2.11 신설)* **지정된 화면(연차 관리 포함)에 한해 열람 전용** — 편집 컨트롤 비활성, 서버 RLS도 조회(SELECT)만 허용 |
| viewer | editor와 동일 화면을 보되 **전면 읽기 전용**. CV Generator·Leave는 **본인 정보만**, **Pipeline 비노출**, **Confidential 마스킹**, Admin·Migration 비노출 (§6.3) |

## 3. 데이터 모델 (요지)

| 테이블 | 주요 컬럼 |
|---|---|
| profiles | id, name, global_role, person_id(→people.id, 매칭키), lpn, status |
| people | id, lpn(unique), name, rank, role, hire_date, termination_date |
| work_items | id, type(project/proposal/pipeline), name, start, main_start, end_date, status(open/closed) — 전 유형, engagement_number, client, hashtags[], description, confidential(bool) (color 미사용, 유형 파생) |
| assignments | id, person_id, kind(work/leave), work_item_id, weekend_dates[], leave_type, start, end_date, note |
| accruals | id, person_id, direction(accrual/usage), type, days(+/-), date, source, note |
| leave_types | name, active(bool), sort_order |
| holidays | id, name, date, recurring |
| settings | key, value (fiscal_year_start_month=7) |
| audit_log | id, user_id, action, target_type, target_id, at |

> 입력 형식: Engagement Code는 기본 `E-00000000`(E- + 숫자 8자리) 형식이며, **예외적으로 `C000000AA` 형식(C + 숫자 6자리 + 영문 대문자 2자리)도 허용**한다(두 형식 중 하나만 만족하면 유효). LPN 숫자 5자리 `00000`. 계정↔인력 매칭은 `profiles.person_id`=`people.id`로만(LPN 미사용).

> **holidays 데이터 소스**: 공휴일(holidays) 데이터는 외부 공공 API(특일 정보 등)에서 연도별로 조회해 채운다. **조회 가능 범위는 최소 2022년부터 현재+1년까지 지원**해야 한다(과거 프로젝트휴가·주말대체 재계산, 퇴사 정산의 소급 계산 등에 2022년 이후 공휴일 데이터가 필요하므로). 현재 2025년부터만 조회되고 있다면, 사용 중인 API가 과거 연도 조회를 지원하는지 확인 후 범위를 확장한다(대부분의 공공 특일 정보 API는 연도 파라미터로 과거 연도도 조회 가능).
> **동기화 버튼 동작(버그 수정 + 단순화)**: 공휴일 화면의 버튼을 **"동기화" 하나로 단순화**한다 — 별도의 "과거 데이터 채우기" 버튼은 불필요하므로 **제거**한다. "동기화" 버튼을 클릭하면 **2022년부터 현재 연도+1년까지의 모든 연도**를 API에서 조회해 holidays 테이블에 일괄 반영한다(현재는 현재 연도+차년도 2개년만 반영되는 제한된 동작으로 되어 있음). **동기화 시 기존에 수동으로 입력해 둔 공휴일은 덮어써지거나 삭제되어도 무방**하다(API 조회 결과로 전체 교체). 동기화 완료 후 결과(반영된 연도 수·공휴일 건수)를 요약해 보여주고 audit_log에 기록한다.

## 4. 색상 체계
- project=파랑, proposal=노랑, pipeline=회색, Leave=녹색. 각 계열 내 자동 음영. 수동 지정 없음. 유형 변경 시 색 자동 전환. **범례 표시는 T-2 참조(Leave는 세부 유형 나열 없이 통합 1항목).**

## 5. 기능 요구사항

### 5.1 실행취소 / 재실행
- Ctrl+Z=실행취소, Ctrl+Y(및 Ctrl+Shift+Z)=재실행. 최대 10단계. 대상: 배정·작업항목·인력·휴가 수동조정. 서버 역연산, 불가 시 안전 실패. editor/admin만. Undo/Redo 버튼.

### 5.2 타임라인 시각화
| ID | 요구사항 |
|---|---|
| T-1 | 인력별/작업별 뷰. 월→주→일 자동 전환. 모든 날짜 MM/DD. 반응형. 막대 라벨 폭 확장 + 호버 툴팁(작업명·고객사·기간, 인력별 뷰는 인력명). |
| **T-2** | 색상: project 파랑 / proposal 노랑 / pipeline 회색 / Leave 녹색. **범례는 4개 상위 카테고리(project/proposal/pipeline/Leave)만 표시한다.** 특히 **Leave는 지정휴가·프로젝트휴가·주말대체·포상·특별·지연보상·리프레시·휴직 등 유형별 세부 색상을 범례에 나열하지 않고, "Leave(녹색 계열)" 한 항목으로만 안내한다**(세부 유형별 음영 차이는 막대·호버 툴팁에서는 유지해도 되나 범례에는 노출하지 않음 — 항목이 너무 많아 가독성이 낮아지는 문제 수정). |
| T-3 | 작업 행은 그 작업 자체의 일정(span)만 표시(인원 막대 미중첩). 인원별 일정은 드롭다운으로. |
| T-4 | 작업 행·'휴가(전체)' 행 드롭다운 펼침/접힘. 작업 내 인원 직급순·이름순. |
| T-5 | 작업항목 더블클릭 → 상세 페이지(§5.7). |
| T-6 | Pre-study 구간 별도 분류·시각 구분(적립·CV 수행기간 제외). |
| T-7 | 인력 칩 드래그 배정(직급 그룹·이름순). **칩 목록에는 재직중 인원과 입사예정 인원을 포함하고 퇴사(P-1 기준) 인원은 제외한다.** 주말·공휴일 음영, 오늘 세로선/버튼, 주말 실근무 마커 날짜 고정, 정렬·필터·FY/기간 필터, **중복 배정 경고(Partner는 참고용 경고만 표시하고 저장 허용, Partner 외 전 직급은 경고 대신 저장 자체를 차단 — E-3a 참조)**, sticky 헤더/라벨. |
| **T-7-TEMP** | ⚠️ **임시 조치(초기 데이터 셋업용, 곧 원복 예정 — 다른 항목과 무관, 이 항목만 별도 처리)**: 타임라인의 FY 필터 옵션을 **FY09부터 FY28까지** 생성한다(§8과 동일한 FY 라벨 규칙 사용, 즉 FY09=2008-07-01~2009-06-30 ... FY28=2027-07-01~2028-06-30). 초기 데이터 이관 작업 때문에 넓은 범위가 필요해 임시로 추가하는 것이며, **이 작업 범위는 FY 필터의 선택 옵션 목록 확장에 한정**한다(다른 타임라인 로직·필터·렌더링·T-7 본문 등 어느 것도 함께 변경하지 않는다). 초기 데이터 셋업이 끝나면 **이 항목(T-7-TEMP)을 PRD에서 삭제하고 FY 필터 옵션 범위를 원래대로(예: 최근 FY 몇 개년) 되돌린다.** |
| **T-8** | **헤더·본문 스크롤 동기화(버그 수정)**: 날짜 캘린더 헤더(상단)와 작업/인력 타임라인 본문(하단)이 **항상 동일한 가로 오프셋으로 함께 스크롤**되어야 한다. 특히 FY 필터로 조회 범위가 바뀌거나 좌우로 스크롤할 때 상·하단이 따로 움직여 날짜와 막대가 어긋나는 문제를 해결한다(공통 스크롤 컨테이너 또는 스크롤 위치 동기화). |
| **T-9** | **FY 다중 걸침 처리**: 한 작업/배정의 기간이 둘 이상의 FY에 걸쳐 있어도, FY 필터·좌우 스크롤 시 헤더 날짜와 막대 위치가 어긋나지 않게 한다(좌표 계산 기준을 단일 viewport 원점으로 통일). |
| **T-10** | **렌더 윈도우 제한**: 기간이 지정된 경우 그 기간의 ±1개월만 렌더링하고, 기간 미지정 시 당일 기준 ±7개월만 렌더링한다(전체 기간을 한 번에 렌더링하지 않아 성능을 확보). 스크롤·필터 변경 시 윈도우를 재계산해 확장한다. |
| **T-11** | **기간 프리셋 버튼**: 이번 주 / 이번 달 / 이번 분기 / 이번 FY 버튼을 제공해 클릭 한 번으로 해당 기간으로 뷰를 이동·필터링한다. |
| **T-12** | **배정 우클릭 컨텍스트 메뉴**: 배정 막대 우클릭 시 편집·복제·삭제·작업 상세 보기·휴가 상세 보기 등의 메뉴를 제공한다. |
| **T-13** | **드래그 중 실시간 미리보기**: 배정을 드래그하거나 리사이즈하는 동안 현재 위치·기간에 해당하는 날짜 범위와 일수를 툴팁으로 실시간 표시한다. |
| **T-14** | **다중 선택 일괄 이동·리사이즈**: 여러 배정을 다중 선택해 한 번에 이동할 수 있다. 선택된 배정들의 종료일 또는 시작일이 동일한 경우 일괄 리사이즈도 지원한다. |
| **T-15** | **Workitem 탭 인력 칩 연동**: Workitem 탭에서 참여 인력을 더블클릭하면 Person(인력별) 탭에서와 동일한 인력 칩 상호작용(선택·하이라이트 등)이 연동된다. |
| **T-16** | **타임라인 진입 시 무조건 오늘 위치로 스크롤(버그 재수정 — 조건부 로직 폐기)**: 기존에는 "대시보드/전역 검색에서 드릴다운으로 들어온 경우에만" 오늘 날짜로 스크롤하도록 조건부로 구현되어 있었으나, 이 조건 분기가 제대로 동작하지 않아(대시보드 이름 클릭 시 여전히 맨 앞 날짜가 보임) 반복적으로 버그가 발생했다. **진입 경로(직접 탭 클릭·드릴다운·전역 검색 등)를 구분하지 않고, 타임라인 탭이 열릴 때는 항상 오늘 날짜 위치로 초기 스크롤**하도록 단순화한다(렌더 완료 후 스크롤 실행). 인력 드릴다운(D-6)의 "해당 인력 행으로 스크롤·하이라이트"는 이 오늘-위치 스크롤과 함께 적용한다(세로 방향은 해당 행, 가로 방향은 오늘 날짜). |

### 5.3 일정 편집
| ID | 요구사항 |
|---|---|
| E-1 | 빈 칸 드래그/더블클릭/칩 드롭으로 생성. 인력별 뷰 빈 칸 클릭 시 해당 인력 자동 선택. |
| E-2 | 작업 선택: status=open 만(전 유형). **인력: 재직중 또는 입사예정 인원만(퇴사 인원 제외, P-1 기준)**. 휴가 유형: active 만. |
| E-3 | 휴가 영업일 스냅. 드래그 겹침 방지 푸시(같은 인력 비중첩). |
| **E-3a** | **직급별 중복 배정 정책(버그 수정)**: **Partner 직급은 동일 기간에 복수 배정(중복 일정)을 허용**한다(여러 프로젝트를 동시에 관장할 수 있으므로). **Partner를 제외한 전 직급(SM/M/Senior/Staff/Intern)은 동일 기간 중복 배정을 하드 차단**한다 — 인력 칩 드래그, 수동 날짜 입력, 칩 드롭 등 배정 생성 경로 어디로도 저장할 수 없다(현재는 T-7의 "중복 배정 경고"만 뜨고 저장 자체는 허용되는 버그가 있어, Partner 이하 직급에 한해 저장 거부로 강화한다). 서버(RPC/트리거)에서도 동일하게 강제한다. |
| **E-5** | **Workitem 기간 자동 확장**: 소속 인력의 투입 **종료일이 workitem 종료일(end_date)보다 늦어지면 workitem의 종료일을 그 인력의 종료일로 자동 연장**한다. 반대로 인력의 투입 **시작일이 workitem 시작일(start)보다 앞서면 workitem의 시작일을 그 인력의 시작일로 자동 앞당긴다**(본 프로젝트 시작 main_start는 그대로 두되 start보다 앞설 수 없다는 무결성 유지). 배정 생성·이동·리사이즈·일괄 조작 어느 경로로든 동일 적용. Closed 작업은 잠금 규칙상 배정 변경 자체가 불가하므로 해당 없음. |
| **E-6** | **배정 리사이즈/기간 변경 시 재검증(버그 수정)**: 배정을 생성한 뒤 드래그로 리사이즈하거나 기간을 늘려 특별휴가 등 잔여 검증 대상 휴가 유형의 사용일수가 증가하는 경우, **신규 생성 시와 동일하게 LV-6(특별휴가 잔여 검증)을 다시 수행**한다(현재는 최초 생성 시에만 검증하고 이후 리사이즈로 늘어난 기간은 검증 없이 저장되는 버그가 있음). 검증에 실패하면 리사이즈를 반려하고 원래 크기로 되돌리며, 부족분을 안내한다. 서버(RPC/트리거)에서도 동일 검증을 강제한다. |
| E-4 | viewer는 모든 편집 컨트롤 비활성(서버 RLS로도 차단). |

### 5.4 인력 관리

### 5.10 휴가 패널 (Leave)
| ID | 요구사항 |
|---|---|
| LV-1 | 적립/사용/잔여, 유형별 소계, 이력(차감 원천), 무급 내역. 사용·잔여 총량 정합(FIFO 원천 표시용). 수동 적립·사용 +/-. 자동 잔여 배정(가장 가까운 빈 영업일, 순서 ①주말대체→②프로젝트휴가→③포상→④지연보상, 유형 구분). |
| **LV-2** | **사람 검색·필터 제공(CV Generator와 동일)**: Leave 탭 상단에 인력 이름 검색 및 직급·재직상태 필터(**재직중/입사예정/퇴사 3단 구분, P-2 기준**)를 두어 대상 인력을 선택해 휴가 상세를 본다. |
| **LV-3** | **viewer 자동 상세창 잠김 버그 수정**: viewer가 Leave 탭에 들어가면 본인 휴가 상세가 자동 표시되되, **상세 창이 모달처럼 화면을 점유하거나 강제로 다시 열려 다른 탭 이동을 막는 일이 없어야 한다.** 닫기/탭 이동이 정상 동작해야 하며, viewer의 Leave는 본인 데이터를 **인라인(탭 본문)으로 표시**하고 강제 모달·자동 재오픈을 하지 않는다. |
| LV-4 | viewer는 Leave에서 본인 데이터만 열람, 편집 기능 비활성. |
| **LV-5** | **휴가 유형별 차감 원천 제한**: 사용 유형이 차감할 수 있는 적립 원천을 다음과 같이 제한한다. **특별휴가 사용은 오직 특별휴가 적립만을 원천으로 차감**한다(프로젝트휴가·포상휴가·주말/휴일대체·지연보상 등 다른 유형의 적립을 원천으로 사용할 수 없다). **지정휴가 사용은 프로젝트휴가·포상휴가·주말/휴일대체·지연보상 적립만을 원천으로 차감**하며(순서는 LV-1과 동일: ①주말/휴일대체→②프로젝트휴가→③포상→④지연보상), **특별휴가 적립은 지정휴가의 원천이 될 수 없다.** **이 제한은 Leave 탭 수동 사용 입력뿐 아니라 타임라인에서 드래그/더블클릭/칩 드롭으로 휴가 배정을 생성하는 경로(§5.3 E-1~E-3)에도 동일하게 적용되어야 하며, 클라이언트 검증만으로는 우회 가능하므로 반드시 서버(RPC/트리거) 단에서도 강제한다.** 자동 FIFO 매칭·수동 사용 입력 모두 이 제한을 따른다. |
| **LV-6** | **특별휴가 입력 단계 검증(버그 수정)**: 특별휴가는 적립이 먼저 존재해야 하며, 사용은 그 시점까지 누적된 특별휴가 적립 잔여 한도 내에서만 입력할 수 있다. **적립 잔여가 부족한 상태에서 특별휴가 사용을 저장하려 하면 입력 단계에서 차단**하고 부족분을 안내하는 에러 메시지를 보여주며, 저장을 허용하지 않는다. 특별휴가는 지정휴가와 달리 **선사용(적립 전 사용)을 허용하지 않는다.** **이 검증은 Leave 탭의 수동 사용 입력 폼과 타임라인에서의 휴가 배정 생성(드래그/더블클릭/칩 드롭) 양쪽 모두에서 동일한 하나의 검증 함수를 공유해 수행해야 하며, 한쪽 경로에서만 적용되고 다른 경로에서는 누락되는 일이 없어야 한다.** 클라이언트 UI 차단과 별개로 **서버(RPC/트리거)에서도 동일 검증을 강제**해 API를 직접 호출해도 우회할 수 없게 한다. |
| **LV-7** | **지정휴가 선사용 표시(버그 수정)**: 지정휴가는 적립이 부족한 상태에서도 사용 입력을 허용한다(선사용). 이 경우 차감 원천 표시를 모호한 "범용 N일" 등의 표기 대신 **"선사용 N일"** 로 명확히 표시하며, AL-7④ '지정휴가 선사용분'과 동일한 FIFO shortfall 기준을 사용한다. |
| **LV-8** | **차감 원천 실시간 재계산(저장값 아님)**: 사용 건의 차감 원천(어느 적립에서 얼마나 차감됐는지, 선사용 여부·일수 포함)은 **저장 시점에 고정·캐시되는 값이 아니라, 조회 시점마다 해당 인력의 LV-5 기준 원천 자격이 있는 전체 적립 이력과 전체 사용 이력을 놓고 처음부터 다시 FIFO 매칭을 재계산**하는 동적 값이다. 예: 지정휴가 사용 시점에는 적립이 부족해 "선사용"으로 표시되었더라도, **이후에(그 적립의 날짜가 사용일보다 나중이어도) 프로젝트휴가 등 유효 원천의 적립이 새로 생기면, 다음 조회부터는 그 새 적립이 우선순위·매칭 규칙에 따라 해당 사용 건의 차감 원천으로 자동 재배정**되어 "선사용" 표시가 사라지고 실제 원천명으로 바뀐다. 이 재계산은 매 화면 조회마다 수행하며 별도 배치·저장 작업으로 하지 않는다(P-1의 실시간 파생과 동일한 설계 원칙). |
| **LV-9** | **수동 적립의 비고 → 원천 표시(조회 시점 폴백, 버그 수정)**: 자동 생성 적립(예: 프로젝트휴가 — work_item에서 자동 산정)이 아니라 **특별휴가·지연보상·포상휴가 등 수동으로 추가하는 적립**은 원천(source)이 work_item으로 자동 연결되지 않으므로, 적립 이력 표의 "원천" 열은 **"source 값이 비어 있으면 note(비고) 값을 그 자리에 대신 표시"** 하는 방식으로 구현한다. **이 치환은 저장 시점에 note를 source 컬럼에 복사해 넣는 방식이 아니라, 화면에 표시할 때마다 매번 계산하는 조회 시점 로직이어야 한다** — 그래야 과거에 이미 입력되어 source가 비어 있는 기존 적립 건도 이 규칙이 적용된 이후 별도 마이그레이션 없이 즉시 올바르게(비고 내용으로) 표시된다. source도 note도 모두 비어 있으면 기존처럼 "—"로 표시한다. |

### 5.13 연차 관리 (Annual Leave, editor/admin 편집 · assistant 열람 전용 탭)

**(A) 퇴사 정산 로직**

핵심 원칙: 팀 제도로 정당 적립된 휴가(프로젝트휴가 등)와 법정연차는 **합산되는 별개 권리가 아니라, 둘 중 더 큰 쪽이 총 휴가 권리**가 된다. 단, **주말/휴일대체 휴가**(실제 근무에 대한 법정 성격의 보상)와 **특별휴가**(예비군·경조사 등 연차와 별개로 발생하는 권리)는 법정연차 쪽 후보에도 함께 더해서 비교한다. 즉 구성원은 max(법정연차 누적 + 주말/휴일대체 누적 + 특별휴가 누적, 팀 정당 적립 누적) 이상의 휴가 권리를 가지며, 그 범위 내 사용은 퇴사 시 불이익이 없다. **무급 휴가는 이 비교의 어느 쪽에도 포함하지 않는다.**

| ID | 요구사항 |
|---|---|
| AL-3 | **총 휴가 권리(Entitlement)** = **max( 법정연차 누적(AL-3a 기준, 보정 포함) + 주말/휴일대체 누적 적립 합 + 특별휴가 누적 적립 합, 팀 정당 적립 합(프로젝트휴가·주말/휴일대체·포상·특별·지연보상 등 Leave 탭의 적립분 전체) )**. **주말/휴일대체·특별휴가는 팀 정당 적립 합에도 이미 포함되어 있으나, 법정연차 쪽 후보 계산에도 별도로 더해서 두 후보값을 비교**한다(어느 한쪽 합계에 두 번 차감되는 것이 아니라, max 비교를 위한 두 후보 각각에 정상적으로 포함되는 것). 법정연차+주말/휴일대체+특별휴가 합이 팀 정당 적립 합보다 크면 그 값을 총 휴가 권리로 한다. **무급 휴가는 적립이 존재하지 않으므로 이 계산에 포함되지 않는다.** |
| **AL-3a** | **퇴사 시 법정연차 재산정(입사일 기준 vs 회계연도 기준, 근로자 불리 금지 원칙)**: **회사 규정상 연차유급휴가는 매년 7월 1일(회계연도 시작일)에 발생하지만, 퇴사 시에는 입사일을 기준으로 연차휴가를 재산정하므로 회계연도 기준 부여일수와 상이할 수 있다**(회사 규정에 명시). 이는 근로기준법상 **회계연도 기준 운영이 입사일 기준보다 근로자에게 불리해서는 안 된다는 원칙**과 부합한다. 따라서 **퇴사 정산 시점에는 AL-2b와 동일한 계산 규칙을 입사일(hire_date) 기준 매년 기준일로 별도 재계산**하고, **[회계연도(7/1) 기준 누적 법정연차]와 [입사일 기준 누적 법정연차] 중 더 큰 값**을 AL-3의 "법정연차 누적"으로 사용한다. 이 입사일 기준 재계산은 별도로 저장하지 않고 **퇴사 정산 조회 시점마다 실시간으로 계산**한다(P-1·LV-8과 동일한 실시간 파생 원칙). |

## 7. 휴가 산정 규칙
1. 프로젝트휴가: **인력×프로젝트 단위 합산 적립** — 한 인력이 같은 프로젝트에 둘 이상의 분리된 배정으로 투입된 경우, 배정별로 각각 round 적립하지 않고 **해당 프로젝트 내 모든 투입 구간(각각 본 프로젝트 구간과의 교집합)의 영업일수 합**(주말·공휴일 제외, §8과 동일한 영업일 정의)을 먼저 구한 뒤 **round(합계 투입 영업일수/10)** 로 1회 적립한다. Pre-study 제외. (예: 12영업일+9영업일로 나뉜 투입 → round(21/10)=2일. 배정별 계산 round(12/10)+round(9/10)=1+1과 다름에 유의.) **(버그 수정) 이 항목은 이전에 "달력일수"로 잘못 기재되어 있었다 — 주말·공휴일을 포함한 전체 일수가 아니라, 영업일수만으로 계산해야 한다.**
2. **주말/휴일대체**: 실근무일만 인정. **주말(토요일·일요일 모두 포함)** 1일당 **0.5일**, **공휴일** 1일당 **1.0일** 적립(pipeline 제외). **주말과 공휴일이 겹치는 날(예: 공휴일이 토·일요일인 경우)은 주말 기준(0.5일)을 우선 적용**하며, 공휴일 가산을 별도로 중복 적용하지 않는다. **(재발 버그 방지) 이 규칙은 타임라인의 "주말 실근무 마커"(T-7) 토글을 포함해, 주말/휴일대체 적립이 발생하는 모든 진입점(마커 토글, 배치 계산, 수동 적립 등)이 동일한 단일 계산 함수를 호출해 적용해야 한다 — 특정 진입점(예: 실근무 마커)만 별도의 하드코딩된 +1일 로직을 갖고 있어 이 규칙을 우회하는 일이 없어야 한다. 일요일 실근무 마커 체크 시에도 예외 없이 0.5일이 적용되는지 반드시 확인한다.**
3. **지연보상(수동 전환)**: 지연보상은 **자동 산정하지 않는다**(기존 자동 판정 로직 제거). 프로젝트 종료 후 휴가 사용 지연이 자의인지 회사 사정인지는 시스템이 판단할 수 없으므로, **관리자가 휴가 패널의 수동 적립으로 '지연보상' 유형을 직접 부여**한다(권장 기준 참고치: 적립분 ≤1:0 / 1.5~3:1 / 3.5~5:2 / 5.5↑:3 — UI 도움말로만 안내). |
4. 사용·잔여 총량 정합(FIFO 원천 표시용, 음수 허용). 수동 +/-.
5. 자동 잔여 배정: §5.10.
6. 휴가 영업일 스냅.
7. **소급 재계산(v2.13 버그 수정 적용)**: 위 규칙2 변경 시점 이전에 "주말+공휴일 겹침일을 공휴일(1.0일)로 과다 적립"했거나 "일요일을 주말로 인식하지 못해 0.5일을 누락 적립"한 기존 accruals 이력을 **일회성 마이그레이션으로 자동 재계산·정정**한다. 별도 관리 화면·미리보기·확인 UI 없이, 배포 시 한 번 실행되는 스크립트가 대상 건을 찾아 규칙2 기준 올바른 값과의 차액만큼 보정(+/-) 항목을 자동으로 적립한다(기존 행은 삭제하지 않고 보존). 사유는 "v2.13 주말/공휴일 겹침 산정 버그 소급 정정"으로 기재하고 audit_log에 기록한다.
